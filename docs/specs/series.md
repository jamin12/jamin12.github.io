# 시리즈 (Series)

> 작성일: 2026-04-06

---

## 1. 역할

시리즈는 **카테고리·태그와 독립적인, 순서가 있는 글 묶음**이다. 같은 시리즈에 속한 글은 "1편 → 2편 → 3편"처럼 흐름이 이어지며, 서로 다른 카테고리에 흩어져 있어도 하나의 연결된 콘텐츠로 묶인다.

---

## 2. 도입 배경

기존에 `기타` 카테고리에 "러너스하이2"라는 스터디 그룹 이름으로 묶여 있던 글들이 있었다. 러너스하이2는 콘텐츠 도메인이 아니라 **활동 단위**였기 때문에, 글이 늘어나면 "이게 뭔 주제지?"를 카테고리명만으로 알 수 없었다.

글들을 내용 기준으로 `개념-정리`/`트러블-슈팅`에 재분류하면서, "이 글들은 순서대로 읽어야 흐름이 이어진다"는 관계를 표현할 수단이 필요해졌다.

### 기존 분류 축의 한계

| 분류 축 | 한계 |
|---------|------|
| **카테고리** (폴더) | 1:1. 글은 하나의 카테고리에만 속함. 순서 없음 |
| **태그** | N:N. 분류용이지 순서 표현 불가 |
| **하위 카테고리** | 태그에서 자동 추론. 작성자의 명시적 의도가 아님 |

시리즈는 이 세 축 어디에도 없는 **작성자가 명시적으로 결정하는 순서 관계**를 담당한다.

---

## 3. 설계 결정: frontmatter vs 중앙 설정 파일

| 기준 | frontmatter (채택) | 중앙 설정 파일 (`series-rules.js`) |
|------|-------------------|----------------------------------|
| 글 추가 시 | `.md` 1개만 수정 | `.md` + 설정 파일 2개 수정 |
| 동기화 위험 | 시리즈명 오타 → 빌드 경고로 잡을 수 있음 | 슬러그 오타 → 조용히 누락 |
| 한눈에 보기 | `posts-meta.json`에서 확인 | 설정 파일 1개 열면 됨 |
| 기존 패턴 | `tags`, `draft`, `cover`와 동일 | `subcategory-rules.js`와 동일 |

**결정 이유**: subcategory는 기존 태그에서 **자동 추론**하는 분류라 중앙 설정이 맞았지만, 시리즈는 작성자가 **명시적으로 결정**하는 관계이므로 글 파일 자체에 선언하는 것이 자연스럽다. 개인 블로그 규모(~70글, 시리즈 3~5개)에서는 중앙 파일 관리 오버헤드가 이점보다 크다.

## 4. Frontmatter 스키마

```yaml
---
title: 기술 선정
date: 2025-12-18
tags: [cdc]
series: CDC 도입기
seriesOrder: 2
---
```

| 필드 | 필수 | 타입 | 용도 |
|------|------|------|------|
| `series` | ⬜ | string | 시리즈 이름. 같은 문자열 = 같은 시리즈. 비어있으면 시리즈 미소속 |
| `seriesOrder` | ⬜ | number | 시리즈 내 순서 (1부터). `series`가 있으면 필수 |

### 검증 규칙 (빌드 타임, 미구현 — 추후 추가 가능)

- `series`가 있는데 `seriesOrder`가 없으면 경고
- 같은 시리즈 내 `seriesOrder` 중복 시 경고
- 시리즈 내 순서에 빈 번호(1, 2, 4)가 있으면 경고
- 시리즈에 글이 1개뿐이면 경고 (오타 가능성)

---

## 5. 빌드 파이프라인

`build-posts-index.mjs`가 `series`와 `seriesOrder`를 `posts-meta.json`에 그대로 전달한다. 추가 파일이나 의존성 없음. 기존 frontmatter 파싱 흐름을 그대로 탄다.

```
.md frontmatter → parseFrontmatter() → build-posts-index.mjs → posts-meta.json
                                         series: "CDC 도입기"
                                         seriesOrder: 2
```

---

## 6. 런타임 조회 (`posts.js`)

| export | 타입 | 용도 |
|--------|------|------|
| `seriesList` | `Array<{ name, count, posts }>` | 모든 시리즈 목록 (최신순). 시리즈 인덱스 페이지용 |
| `getPostsBySeries(name)` | `Post[]` | 특정 시리즈의 글 목록 (seriesOrder 순) |
| `getSeriesNav(slug)` | `{ series, current, total, prev, next, all }` | 현재 글의 시리즈 내 위치 + 이전/다음 글. 글 상세 네비게이션용 |

### `getSeriesNav` 반환 예시

```js
{
  series: 'CDC 도입기',
  current: 2,        // 현재 2번째
  total: 5,          // 전체 5편
  prev: { slug: '주제찾기', title: '주제 선정', ... },
  next: { slug: 'poc-및-피드백', title: 'Poc 및 피드백', ... },
  all: [ ... ]       // 전체 시리즈 글 배열 (seriesOrder 순)
}
```

## 7. 현재 시리즈 (2026-04-06 기준)

| 시리즈 | 편수 | 카테고리 분포 |
|--------|------|-------------|
| CDC 도입기 | 5 | 개념-정리 (5) |
| 모니터링 구축기 | 7 | 개념-정리 (4) + 트러블-슈팅 (3) |
| CI 개선기 | 2 | 개념-정리 (1) + 트러블-슈팅 (1) |

모니터링 구축기와 CI 개선기는 **카테고리를 넘나드는 시리즈**의 실제 사례다. 개념 설명 글은 개념-정리에, 에러 해결 글은 트러블-슈팅에 있지만 시리즈로 순서가 이어진다.

## 8. UI 설계

### 8.1 글 상세 페이지 (최우선)

시리즈 글을 읽는 독자가 **"이 글은 시리즈의 몇 번째이고, 다음 글은 뭔지"**를 자연스럽게 인지해야 한다.

**상단 시리즈 배너** — 글 제목 위(kicker 영역)에 시리즈명 + 현재 위치 표시

```
CDC 도입기  ·  2 / 5
───────────────────────
기술 선정
2025-12-18  ·  5분
```

클릭 시 시리즈 전체 목차를 아코디언으로 펼치는 것도 고려 가능.

**하단 이전/다음 네비게이션** — 글 본문 아래, 기존 "목록으로" 영역을 확장

```
┌──────────────────────────────────────────┐
│  ← 이전                        다음 →    │
│  1. 주제 선정            3. PoC 및 피드백  │
└──────────────────────────────────────────┘
```

### 8.2 글 목록 카드 (2차)

카드에 시리즈 라벨을 작게 표시해 시리즈물임을 한눈에 인지

```
[CDC 도입기 2/5]  기술 선정
```

### 8.3 시리즈 인덱스 (3차)

홈 또는 별도 페이지에서 시리즈 단위로 묶어 보여주는 뷰. `seriesList` export가 데이터를 제공.

---

## 9. 새 시리즈를 추가하려면

1. 시리즈에 포함할 글의 frontmatter에 `series`와 `seriesOrder`를 추가
2. 같은 `series` 문자열을 사용하면 자동으로 묶임
3. 카테고리는 글 내용에 맞게 자유롭게 배치 (시리즈와 무관)
4. 빌드 시 자동으로 `posts-meta.json`에 반영

---

## 10. 연관 도메인

| 도메인 | 관계 |
|--------|------|
| **Content** | frontmatter 스키마 확장 (`series`, `seriesOrder`). 빌드 파이프라인에서 메타 JSON에 주입 |
| **Routing** | 글 상세 페이지에서 `getSeriesNav(slug)` 소비. 시리즈 인덱스 페이지 라우트 추가 가능 |
| **Layout** | PostDetail에 시리즈 배너 + 이전/다음 네비 컴포넌트 추가 |
