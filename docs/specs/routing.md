# 라우팅 (Routing)

> 작성일: 2026-04-05

---

## 1. 역할

URL과 화면을 매핑하는 계층이다. 사용자가 글 목록을 탐색하고, 특정 글·카테고리·태그로 이동하고, 없는 페이지에서 404를 만나는 흐름을 관리한다.

---

## 2. 라우트 구조

| 경로 | 화면 | 내용 |
|------|------|------|
| `/` | PostList | 4섹션 대시보드 (01 최신 글 · 02 카테고리 · 03 카테고리별 최근 · 04 태그). 상세는 [layout.md](layout.md#postlist-홈-) |
| `/posts/:slug` | PostDetail | 글 본문 |
| `/categories/:category` | CategoryPage | 해당 카테고리 글 목록 — 하위 그룹이 정의돼 있으면 group별 섹션으로 분할 |
| `/categories/:category/:subcategory` | CategoryPage | 해당 카테고리 안의 특정 하위 그룹만. 같은 컴포넌트, URL 파라미터로 필터링 |
| `/tags/:tag` | TagPage | 해당 태그 글 목록 |
| `*` (기타) | NotFound | 404 |

### 하위 카테고리 URL 설계 (2026-04 4차)

카테고리 안의 논리적 하위 주제(network, elasticsearch, 러너스하이2 주제1/2/3 등)는 물리 폴더가 아니라 **태그 기반 런타임 그룹**으로 구현된다 ([content.md §8의 "옵션 D의 보완"](content.md) 참조). URL은 nested path 방식을 채택.

**대안과 기각 이유**

| 후보 | 기각 이유 |
|------|----------|
| (A) `/categories/:cat?sub=network` 쿼리 파라미터 | 북마크·공유에 불리. SPA 라우터에서 쿼리 바뀔 때 컴포넌트 재마운트 제어도 추가 비용 |
| (B) `/sub/:cat/:sub` 완전히 분리된 경로 | 부모 카테고리와의 관계가 URL에서 끊김. 뒤로가기 UX 복잡 |
| (C) `/categories/:cat/:sub` nested | 부모·자식 계층이 URL 구조에 그대로 드러남. 라우트 하나 추가로 구현 가능. **채택** |

React Router v7에서 라우트 두 개를 등록(`/categories/:category` + `/categories/:category/:subcategory`)하고 둘 다 `CategoryPage` 컴포넌트로 매핑. 컴포넌트 내부에서 `useParams()`로 `subcategory` 유무를 판정해 전체 뷰 vs 단일 하위 뷰를 분기.

### 확장 예정 (2차)

| 경로 | 화면 |
|------|------|
| `/about` | About 페이지 |
| `/search?q=...` | 검색 결과 |

## 3. `/posts/:slug`가 카테고리를 URL에 안 넣는 이유

### 후보

- (A) `/posts/:slug` - 카테고리 없음
- (B) `/posts/:category/:slug` - 카테고리 포함

### 선택: A

| 이유 | 설명 |
|------|------|
| slug가 전역 유일 | 카테고리 없이도 식별 가능 |
| 카테고리 이동에 강함 | 글을 다른 폴더로 옮겨도 URL 안 깨짐 (외부 링크·북마크 보존) |
| URL이 짧고 읽기 쉬움 | 공유·인쇄 친화 |
| 카테고리는 탐색 축일 뿐 | 글의 정체성은 slug, 카테고리는 필터 |

### slug 중복은 어떻게

Content 파이프라인에서 빌드 타임에 slug 중복을 검사하고, 있으면 경고를 띄운다. 중복이 있으면 같은 slug가 뒤에 온 파일로 덮어쓰여지므로 실제로는 하나만 접근 가능.

## 4. 라우터 선택

### 후보

| 후보 | 장점 | 단점 |
|------|------|------|
| **React Router v7** | 표준, 풍부한 기능, 생태계 | 번들 크기 |
| **wouter** | 초경량 (1KB) | 기능 제한, 스크롤 복원 등 직접 |
| **직접 구현** | 의존성 0 | 엣지 케이스 부담 |

### 결정: React Router v7

이유:
- SPA 라우팅의 디테일(스크롤 복원, 앵커 네비, 데이터 로더 등)을 직접 다루기엔 학습 목적 외에 얻는 게 적다
- v7는 React 19와 잘 맞고 data loader API가 블로그 정적 데이터와 궁합이 좋다
- 번들 크기는 블로그 전체 규모에 비하면 미미
- View Transitions API와의 통합이 공식 지원

## 5. 한글 slug 처리

### 문제

파일명·폴더명에 한글이 가능하므로 URL도 한글이 들어올 수 있다.

```
/posts/리액트19-use-훅
/categories/네트워크
```

브라우저 주소창에서는 한글이 디코딩되어 보이지만, 실제 전송은 `%EB%A6%AC%EC%95%A1%ED%8A%B8...`처럼 퍼센트 인코딩된다.

### 처리 규칙

| 단계 | 처리 |
|------|------|
| 링크 생성 | `encodeURIComponent(slug)` 명시 적용 |
| 라우트 매칭 | React Router `useParams()`는 자동 디코딩 |
| Post 조회 | 디코딩된 slug로 Post 배열에서 `find` |
| 화면 표시 | 디코딩된 원문 그대로 노출 |

### 왜 영문 slug로 강제하지 않는가

- 파일명 = URL 대응 원칙. 파일명을 영문으로 강제하면 한글 제목의 글을 영문 slug로 또 관리해야 함 (이중 관리)
- 현대 브라우저는 한글 URL을 문제없이 보여줌
- 검색 엔진도 한글 URL을 허용한 지 오래

### 파일명 작성 규약

| 금지 | 이유 |
|------|------|
| 공백 | URL에서 `+` 또는 `%20`으로 변환되어 혼란 |
| 대소문자 혼용 | 일부 서버/CDN이 대소문자 구분 |
| 점(`.`), 슬래시(`/`) | URL 예약 문자 |
| `?`, `#`, `&` | URL 파라미터 문자 |

단어 구분은 하이픈(`-`) 사용. 한글 + 하이픈 조합은 허용.

## 6. 404 처리

### 대상

- 존재하지 않는 slug: `/posts/없는글`
- 존재하지 않는 카테고리/태그
- 완전히 정의되지 않은 경로: `/foo/bar`

### 표시 내용

- "이 글을 찾을 수 없어요" 메시지
- 홈으로 돌아가기 버튼
- 최근 글 3개 제안 (빈손으로 돌려보내지 않기)

### SPA 404의 특수성

GitHub Pages에 SPA를 배포하면 `/posts/foo`로 직접 접속 시 서버가 진짜 404를 반환한다. 이를 우회하기 위한 `404.html` 폴백은 [deployment.md](deployment.md)에서 다룬다.

## 8. View Transitions (선택)

React 19 + React Router v7은 View Transitions API를 기본 지원한다.

### 기대 효과

- 글 목록 → 상세: 카드 제목이 본문 제목으로 자연스럽게 연결
- 카테고리 간 이동: 목록 컨테이너가 부드럽게 전환

### 적용 범위

초기엔 **글 목록 ↔ 상세** 하나만. 과도한 전환은 오히려 방해.

CLAUDE.md에 `vercel-react-view-transitions` 스킬이 있으므로 실제 구현 단계에서 이 스킬을 호출한다.
