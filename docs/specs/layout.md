# 레이아웃/UI (Layout)

> 작성일: 2026-04-05

---

## 1. 역할

사이트의 **공통 프레임**(헤더·푸터·본문 컨테이너)과 **페이지별 레이아웃**(목록 카드·상세 본문·카테고리 인덱스)을 정의한다.

블로그는 읽기 경험이 전부이므로, 이 계층의 목표는 단 하나: **본문이 가장 잘 읽히도록 주변 요소를 조용히 만드는 것**.

---

## 2. 디자인 원칙

### 1. 본문 우선

헤더·푸터·사이드바는 글을 방해하지 않는다. 색·크기·애니메이션이 본문보다 튀어선 안 된다.

### 2. 타이포그래피가 곧 디자인

개발 블로그는 텍스트가 90%다. 폰트·자간·행간·색 대비가 전부 결정한다. 화려한 일러스트·그라데이션보다 **읽기 좋은 기본 텍스트**가 우선.

### 3. 제네릭 AI 룩 금지

CLAUDE.md에 적힌 `frontend-design` 원칙대로, 모든 AI 생성물이 수렴하는 "안전한 현대적 룩"(회색 + 파란색 accent + 둥근 카드 + 가벼운 그림자)을 피한다. 개인 블로그답게 개성이 있어야 한다.

### 4. 느리게 만들지 않기

스타일 비용이 읽기 속도를 해쳐선 안 된다. 애니메이션·그림자·블러는 목적 있을 때만.

### 5. 브랜드 방향: Dev Dashboard Editorial (2026-04 확정, 4차 반복)

§2.3의 "제네릭 AI 룩 금지" 원칙을 **실행 방향**으로 구체화했다. 선택한 길: **Dev Dashboard Editorial — 사이드바 셸 + 카드 그리드 구조 + Refined Editorial Dark 문자(serif · Electric Blue)**.

4차는 3차(Refined Editorial Dark)의 **캐릭터(Fraunces + Inter, Blue accent, 섹션 번호)** 는 그대로 가져가고, 홈의 **구조**만 사이드바 + 히어로/그리드형 대시보드 레이아웃으로 교체했다. 레퍼런스: Inpa Dev(구조) + impeccable.style(editorial 문법) + Linear(다크 미니멀).

| 축 | 값 |
|----|-----|
| 성격 3단어 | **Sharp · Honest · Playful** (3차와 동일) |
| 핵심 원리 | "editorial 캐릭터는 유지하되 홈의 구조를 대시보드로 열어 탐색성을 끌어올린다" |
| 셸 | 왼쪽 260px sticky 사이드바(프로필·카테고리 트리·테마 토글) + 오른쪽 메인. `<1024px`에선 사이드바 숨기고 top bar로 대체 |
| 홈 구조 | `01 최신 글`(1 히어로 + 3 side, SplitLatest) → `02 카테고리`(CategoryGrid) → `03 카테고리별 최근`(기존 section loop) → `04 태그` |
| 색 전략 | 3차와 동일 — 3층 배경 + 텍스트 4단 투명도 + Electric Blue 포인트. 카테고리별 arbitrary 색 매핑 **금지** (editorial 절제 유지) |
| 카테고리 시각 | 카테고리 이름의 **첫 글자**(`getCategoryLetter()`)를 Fraunces serif로 letter-mark 처리 + 공통 `--bg-subtle` 카드. 랭킹 배지 #1만 accent 색, #2·#3은 `--text-muted`. **이모지·이모티콘은 일체 사용하지 않음** (CLAUDE.md 전역 규칙: 사용자가 명시적으로 요청할 때만) |
| 커버 이미지 | frontmatter `cover: ./images/foo.png` (optional, `./images/` 규칙은 본문과 동일). 없으면 `--bg-subtle`→`--bg-hover` 그라데이션 + 카테고리 letter-mark fallback |
| 사이드바 아바타 | 정사각형 4px radius + Fraunces "J" letter-mark. 실제 이미지 없음 (의존성 0) |
| 유지한 3차 자산 | Fraunces + Inter + Noto Serif KR 스택, CSS counter 번호, `--accent: #2b5bff`, 본문 17px/1.8, 다크 기본, shiki 듀얼 테마 |

**왜 3차를 또 바꾸는가** (같은 날 4차 반복):
- 3차는 타이포·컬러 discipline은 완성도가 높았으나 **홈의 구조가 "단일 컬럼 섹션 나열"**이라 카테고리 탐색성이 약했다. 사용자가 들어왔을 때 "이 블로그에 뭐가 있는지"를 한눈에 잡기 어려움
- Inpa Dev 스타일의 **사이드바 + 카드 그리드 + 히어로 포스트** 구조는 "읽으러 온 독자"보다 "둘러보러 온 방문자"에게 훨씬 친화적
- 해결책: 3차의 **문자**(서체·색·절제)는 100% 유지하고 **구조**만 열어서 둘을 병립시킨다. 캐릭터 손실 없이 탐색성만 얻는 구조

**카테고리별 색상을 거부한 이유**: 레퍼런스 Inpa Dev가 알록달록한 건 AWS·Linux·Node 같은 **브랜드 컬러**가 이미 존재하기 때문. 한국어 카테고리명(`개념-정리`, `코테` 등)은 브랜드 색이 없고 임의 배정은 "회색 + 파랑" 제네릭 룩으로 회귀할 위험. 카테고리의 시각적 구분은 **이름 첫 글자의 letter-mark**(예: `개념-정리` → `개`, `코테` → `코`) 하나로 족하고, 나머지 UI는 Electric Blue 단일 액센트 규칙(`.impeccable.md` Design Principle)을 유지

**이모지/이모티콘 일체 금지**: 4차 작업 중 초안에서 카테고리 아이콘으로 컬러 이모지(📚🧩🔧✨)를 사용했다가 사용자가 "이모티콘을 빼줘"로 기각. 전역 CLAUDE.md 규칙("사용자가 명시적으로 요청할 때만 이모지 사용")을 어긴 기본값 오류. 교체안으로 **Fraunces serif 첫 글자 letter-mark**를 채택. 이 letter-mark 패턴은 사이드바 브랜드 아바타의 `J` 마크와 같은 시스템을 공유해 시각 일관성도 확보

**기각 기록 — 2026-04-05 하루 동안 네 번의 반복**

1. **1차: Polished Brutalism (기각)** — 2px 하드 테두리 + `6px 6px 0 #2b5bff` 하드 섀도우. Gumroad 옛 기억 오해, cards-on-cards 안티패턴
2. **2차: Refined Minimal (기각)** — Inter 단일, 카드 전부 투명, 섹션 번호 제거. editorial 캐릭터까지 사라진다는 피드백
3. **3차: Refined Editorial Dark** — 1·2차를 합친 타협점. Fraunces + Inter 이중 스택, CSS counter, 얇은 카드 보더. **타이포·색 discipline**은 여기서 완성됨
4. **4차 (현재): Dev Dashboard Editorial** — 3차의 문자는 유지, 홈의 **구조**를 사이드바 + 히어로 + 그리드로 확장. 탐색성 확보

**낡은 결정을 지우지 않는 이유** (CLAUDE.md 규칙 #6): 네 번의 실패/성공 기록이 모두 보존되어야 다섯 번째 번복 요청이 와도 즉시 맥락을 복원할 수 있다. 특히 "카테고리별 색 매핑"은 미래에 다시 제안될 가능성이 높은 아이템이므로 기각 이유를 명시적으로 남김

상세 원칙·팔레트·안티 레퍼런스는 프로젝트 루트의 [`.impeccable.md`](../../.impeccable.md)에 영구 저장.

### 5.1 (Archive) 3차 반복: Refined Editorial Dark

§2.3의 "제네릭 AI 룩 금지" 원칙을 **실행 방향**으로 구체화한 3차 반복. 4차에서 **구조만** 교체되었고 **문자(타이포·색·절제)** 는 그대로 계승되었으므로 아래 팔레트·폰트 결정은 여전히 유효하다.

| 축 | 값 |
|----|-----|
| 성격 3단어 | **Sharp · Honest · Playful** |
| 핵심 원리 | "장식은 줄이되 editorial 캐릭터는 유지". 미니멀과 캐릭터 없음은 다르다 |
| 색 전략 | 3층 배경(`#0a0a0b` · `#111116` 카드 · `#14141a` hover), 텍스트 다층 투명도, 테두리 `#1f1f24`/`#2b2b33` |
| 포인트 | Electric Blue `#2b5bff` — **딱 네 곳에만** (링크·포커스·인라인 코드·hover 제목) |
| Display 폰트 | **Fraunces** variable serif (`opsz,wght@9..144,400..700`) + **Noto Serif KR** 한글 fallback — impeccable.style의 editorial serif 에너지 |
| Body 폰트 | **Inter** 400/500/600/700 + Pretendard Variable 한글 fallback |
| 섹션 번호 | CSS 카운터로 카테고리 앞에 `01`, `02`, `03` 자동 부착 (impeccable.style의 "01 The Antidote" 패턴) |
| 카드 | 얇은 `1px solid var(--border)` + `--bg-subtle` 배경 틴트 + `border-radius: 10px`. Hover 시 배경·보더 한 단계 lift |
| 참조 결 | **impeccable.style** (일순위: 서체·섹션 번호·editorial 문법) + Linear (다크 기준점·거대 제목) + 현재 Gumroad (얇은 카드 보더) |

상세 원칙·팔레트·안티 레퍼런스는 프로젝트 루트의 [`.impeccable.md`](../../.impeccable.md)에 영구 저장. 재디자인 관련 모든 `teach-impeccable` 기반 스킬이 그 파일을 전제로 동작한다.

**왜 이 방향**: 2026-04 이전의 `src/App.css`는 §2.3이 금지한 "제네릭 AI 룩"에 근접했다 — 회색 + 파란 accent + 둥근 카드 + 가벼운 soft shadow. 그 반대 방향을 레퍼런스에서 찾았고, **impeccable.style이 editorial serif·섹션 번호·얇은 카드·핑크(→블루) 액센트로 구현한 패턴**이 가장 근접한 해였다. 개성은 장식이 아니라 **editorial 문법** 레이어에 녹인다.

**기각 기록 — 같은 날(2026-04-05) 두 번 틀렸다가 세 번째에 맞춤**

1. **1차 시도: Polished Brutalism (기각)**
   - 2px 하드 테두리 + `6px 6px 0 #2b5bff` 오프셋 섀도우, Space Grotesk 디스플레이, 브루탈 박스 체이닝
   - 기각 이유: Gumroad 2019~2022 브루탈 시절 기억이 낡아 오해. 현재 Gumroad는 이미 미니멀로 리뉴얼됨. Linear도 no heavy shadows, impeccable.style은 cards-on-cards를 명시적으로 경고. 방향 자체 어긋남

2. **2차 시도: Refined Minimal (기각)**
   - 카드 보더 전부 제거, 배경 투명, Inter 단일 폰트 패밀리, 섹션 번호·키커 없음, 위계는 크기로만
   - 기각 이유: 사용자가 레퍼런스 이미지 5장을 보여주며 "너무 다르다"고 피드백. **"미니멀"을 "캐릭터 없음"으로 해석한 게 오류**. 실제 레퍼런스들은 editorial 캐릭터(서체·섹션 번호·얇은 카드)를 유지하면서 장식성만 제거. 2차는 캐릭터까지 제거

3. **3차 (현재): Refined Editorial Dark** — 1차의 캐릭터 욕망 + 2차의 restraint를 동시에. 서체는 impeccable.style에서, 카드는 Gumroad에서, 레이아웃·크기·여백은 Linear에서 흡수

**낡은 결정을 지우지 않는 이유** (CLAUDE.md 규칙 #6): 번복 이유가 다음 번복을 막는다. 1차·2차 실패의 구체적 원인이 보존되어야 미래에 "어, 브루탈?" / "어, 더 스트립?" 요청이 나올 때 즉시 "이미 시도했고 왜 실패했는지"를 확인할 수 있다.

**이전 방향 (2026-04 리디자인 전)**: radius 6~10px 둥근 카드, 1px 얇은 보더, `transition` 중심 hover, 라이트 기본. 제네릭 룩으로 수렴한다는 판단으로 교체됨.

## 3. 공통 프레임

```
┌────────────────────────────────┐
│  Header (사이트 타이틀, 네비)    │
├────────────────────────────────┤
│                                │
│       본문 영역                  │
│       (최대 너비 제한)           │
│                                │
├────────────────────────────────┤
│  Footer (GitHub, 외부 링크)      │
└────────────────────────────────┘
```

### Header

- 왼쪽: 사이트 타이틀 (홈 링크)
- 오른쪽: 네비 (글 목록, 카테고리, About), 다크모드 토글
- 높이 최소화 (글 읽을 땐 거의 없는 듯)
- 스크롤 동작(숨김 vs sticky)은 구현 단계에서 결정

### Footer

- GitHub, 연락처, 라이선스 한 줄
- 글 끝에서만 보이면 충분

### 본문 너비 (2026-04-05 4차에서 두 번 조정)

| 환경 | 최대 너비 | 이유 |
|------|---------|------|
| 데스크톱 (≥ 1280px) | **880px** (본문) + 260px (TOC), post-layout 총 1240px | 사이드바 셸 환경에서 본문 720 → 820 → 880으로 단계적 확장. 한 줄 ~75자로 60~75자 가이드라인 상단 한계 |
| 태블릿 (≥ 640px) | 880px (TOC 없음) | 좌우 여백 2rem |
| 모바일 (< 640px) | 화면 꽉 (좌우 패딩 1.25rem) | 가로 공간 희소 |
| `.app__main--post` 컨테이너 캡 | **1480px** (`<1480` 에서만 centered), 그 외 페이지는 1100px | post 페이지만 예외적으로 넓게 |

**수치 변경 기록 (2026-04-05, 같은 날 두 차례)**

1. **1차 조정 (720 → 820)**: 4차 사이드바 셸 전환 후 사용자 피드백 "본문이 가운데로 몰려 있음". 1918px 모니터에서 사이드바(260px) 오른쪽에 280px+ 빈 공간이 생기는 문제. `app__main` 1100 캡 + `post-layout` 1000 캡 + 본문 720 캡이 3중 중심 정렬된 결과. 해결: post 전용 `.app__main--post` 1400px, post-layout 1160px, 본문 820px, TOC 220→240, 브레이크포인트 1200→1240
2. **2차 조정 (820 → 880)**: 1차 적용 후 사용자 "좀 더 넓어도 될 거 같아". 820px는 ~68자(권장 범위 중상단)였고, 가이드라인 상한 75자까지 여유가 있었음. 해결: 본문 820→880, TOC 240→260, post-layout 1160→1240, `.app__main--post` 1400→1480, 브레이크포인트 1240→1280

**현재 880px의 한 줄 한국어 글자 수**: 약 **75자** — `.impeccable.md`의 60~75자 가이드라인 **상한**. 이보다 넓히면 시선 복귀(saccade return) 거리가 길어져 독서 속도가 눈에 띄게 떨어진다는 타이포그래피 연구 결과. **3차 조정이 있다면 이 한계를 명시적으로 검토해야 함**

**이전 이전 수치 (2026-04-05 오전, 3차)**: 본문 720px · TOC 220px · post-layout 1000px · 브레이크포인트 1200px. 센터드 컬럼이 사이드바 없는 구조에 최적화돼 있던 3차 구조의 잔재. 4차 사이드바 셸에서 시각 균형이 깨져 두 차례에 걸쳐 확장됨. 원칙은 유지 — "본문은 읽기가 전부", 하지만 컨테이너와의 시각 균형도 "읽기 경험"의 일부라는 교훈

## 4. 페이지별 레이아웃

### PostList (홈, `/`) — 2026-04-05 4차: Dev Dashboard 구조

홈은 **4개 섹션 수직 적층**으로 구성한다. 각 섹션은 CSS counter로 `01`/`02`/`03`/`04` 자동 번호가 붙고, Fraunces serif 제목 + 작은 서브타이틀 패턴을 공유한다.


**섹션별 규칙**

| 번호 | 섹션 | 내용 | 컴포넌트 |
|------|------|------|---------|
| 01 | 최신 글 | 최신 4개. 좌측 히어로 카드 1개 + 우측 compact 카드 3개 (비대칭 split) | `SplitLatest` → `HeroPost` + `SideCard` |
| 02 | 카테고리 | 모든 카테고리를 아이콘 카드로. 글 수 기준 상위 3개에 #1/#2/#3 랭킹 배지 | `CategoryGrid` |
| 03 | 카테고리별 최근 | 카테고리별 최신 `POSTS_PER_CATEGORY`(기본 5)개. 기존 `CategorySection` 재사용 | `PostCards` loop |
| 04 | 태그 | 사용 빈도 내림차순 chip | `TagFooter` |

**왜 "그 외" 칩 블록을 삭제했는가** (3차에서 있던 요소): 02 CategoryGrid가 이미 **모든** 카테고리를 시각 카드로 노출하므로 하단에 같은 정보를 텍스트 칩으로 다시 보여줄 필요가 없어졌다. 태그만 하단에 남김.

**숫자 조정**은 여전히 `src/pages/PostList.jsx` 상단 상수(`POSTS_PER_CATEGORY`). `MAX_CATEGORY_SECTIONS`는 CategoryGrid가 모든 카테고리를 다 보여주는 방향이 되면서 03 섹션에선 무제한이 되어 제거(현재 카테고리 4개라 제한 의미 없음. 5개 초과하면 재도입).

**대안과 기각 이유**

| 후보 | 기각 이유 |
|------|----------|
| (A) 상단 상단에 Big Hero 1개 + 그 아래 카테고리 그리드 | Hero와 SplitLatest 히어로가 중복. SplitLatest 좌측 카드가 이미 Big Hero 역할 |

**카테고리 이름 표시**: URL과 파일시스템엔 하이픈(`개념-정리`), 화면엔 공백(`개념 정리`). `formatCategory()` 헬퍼가 `posts.js`에 있고, 카드·섹션 헤더·상세 kicker·카테고리 페이지 타이틀에서 사용. 태그는 변환 안 함(영문 `spring-boot` 같은 것은 하이픈 유지).

## 5. 타이포그래피

### 폰트 스택 (2026-04 3차 반복: Refined Editorial Dark)

| 역할 | CSS 변수 | 스택 |
|------|---------|------|
| Display (제목·큰 타이포) | `--font-display` | **Fraunces** variable serif (`opsz,wght@9..144,400..700`) → **Noto Serif KR** 한글 → Pretendard → Georgia → serif |
| Body·UI | `--font-sans` | **Inter** 400/500/600/700 → Pretendard Variable 한글 → 시스템 sans |
| 코드 | `--font-mono` | JetBrains Mono, D2Coding, Consolas |

**이중 스택 전략 (3차 방향)**. 제목은 serif, 본문은 sans. 이 대비가 editorial 사이트의 핵심 문법이고, 1·2차 시도에서 놓친 **캐릭터**를 이 층에 담는다.

**Fraunces** (제목 디스플레이):
- Google Fonts의 variable serif. `opsz` (optical size 9~144), `wght` (400~700) 2축을 지원
- `font-optical-sizing: auto`로 크기별 최적 렌더링 — 큰 제목(48~56px)에선 우아하게, 작은 UI 텍스트에선 또렷하게
- impeccable.style의 "The Antidote"·"The Framework" 같은 editorial serif 에너지를 구현
- Latin 글리프만 포함 → 한글은 Noto Serif KR로 자동 fallback
- `.site-title`, `.post-list__title`, `.category-section__title`, `.page-header__title`, `.post__title`, `.post__body h1~h4`, `.not-found__code`, 그리고 `<blockquote>` 본문에 사용

**Noto Serif KR** (한글 제목 fallback):
- 한글 글리프를 serif 계열로 제공해 Fraunces와 시각적 일관성 유지
- "네트워크 기초" 같은 한글 제목이 sans(Pretendard)로 떨어지면 Fraunces serif와 섞여 어색하므로, 가중치 400~700 variable로 함께 로드
- 파일 사이즈 우려가 있지만 Google Fonts subsetting + 지연 로딩으로 FCP에 영향 없음

**Inter** (본문·UI):
- Linear, Vercel, impeccable.style 등 레퍼런스 다수가 Inter 기반
- `cv11`·`ss01`·`ss03` OpenType 기능을 `html`에 적용해 single-story a, open digits, slashed zero 등 디테일 확보
- 한글은 Pretendard Variable로 자동 fallback (Pretendard 자체가 Inter metrics에 맞춰 설계됨)

**본문 수치**: 17px · line-height 1.8 · `word-break: keep-all`. 한국어는 자소 밀도가 높아 영문보다 약간 크게, 행간 넉넉히. 이 수치는 **절대 양보하지 않는다** (.impeccable.md Design Principle #6).

**이전 스택 기록**
- **~ 2026-04 (리디자인 전)**: Pretendard 단일 (제목·본문 동일 패밀리)
- **2026-04-05 1차 Polished Brutalism (기각)**: Space Grotesk + Pretendard 이중 스택. 방향 자체가 기각되면서 다음 시도에서 폐기
- **2026-04-05 2차 Refined Minimal (기각)**: Inter 단일 패밀리, 제목·본문 통일. "단일 폰트로 위계를 만든다"가 의도였으나 **editorial 캐릭터가 사라진다**는 피드백을 받고 3차에서 Fraunces serif를 추가해 이중 스택 복원

| 요소 | 크기 (desktop) | 행간 |
|------|--------------|------|
| h1 | 2.25rem | 1.25 |
| h2 | 1.75rem | 1.3 |
| h3 | 1.375rem | 1.4 |
| 본문 | 1.0625rem (17px) | 1.75 |
| code (inline) | 0.95em | - |
| code block | 0.9rem | 1.6 |

**본문 17px + 행간 1.75**: 한국어는 자소 밀도가 높아 영문보다 약간 크게, 행간도 넉넉히. 16px는 한국어에서 다소 답답.

### 색 대비

- 본문 텍스트 / 배경 대비: WCAG AA 이상 (4.5:1)
- 링크는 밑줄 또는 색으로 구분 (색만 의존 금지 — 색맹 접근성)

## 7. 다크 모드

### 전환 방식 (2026-04 업데이트: 다크 기본으로 플립)

- **다크가 기본**. 시스템이 `prefers-color-scheme: light`를 명시할 때만 라이트로 부팅
- 수동 토글 버튼 (헤더, `.theme-toggle`)
- 선택은 `localStorage`에 저장 → 이후 방문은 마지막 선택 우선
- FOUC 방지 인라인 스크립트는 React 마운트 전에 `<html data-theme>`을 설정 (`index.html`)

**바뀐 이유**: Polished Brutalism 방향에선 다크가 **정체성**이다. 블루 그림자(`box-shadow: 6px 6px 0 #2B5BFF`)·흰색 테두리 반전 트릭·코드블록 좌측 블루 스트라이프가 모두 다크 베이스를 기준으로 설계됨. 개발자·긴 기술 글 독자는 다크 선호가 압도적이기도 함. 라이트는 보조 모드로 유지(토큰은 그대로 정의됨).

**이전 방식 기록**: 2026-04 이전엔 시스템 설정을 그대로 따랐고 `prefersDark` 체크 → 기본 라이트. 이 파일(§2.5) 참조.

### 색 설계 (2026-04 Refined Minimal 팔레트)

| 요소 | 다크 (기본) | 라이트 (보조) |
|------|----------|------------|
| 배경 (`--bg`) | `#0a0a0b` | `#fafafa` |
| 서브 배경 (`--bg-subtle`) | `#101014` | `#f2f2f4` |
| 본문 텍스트 (`--text`) | `#f5f5f7` | `#0a0a0b` |
| 부제·요약 (`--text-secondary`) | `#c4c4cc` | `#3a3a40` |
| 메타 (`--text-muted`) | `#8b8b93` | `#6b6b73` |
| 희미 (`--text-faint`) | `#5b5b63` | `#9b9ba3` |
| 주 테두리 (`--border`) | `#1f1f24` | `#e8e8ec` |
| 약한 테두리 (`--border-muted`) | `#141418` | `#f2f2f5` |
| 코드 배경 (`--code-bg`) | `#121216` | `#f2f2f5` |
| 코드블록 배경 (`--pre-bg`) | `#101014` | `#f6f6f8` |
| 포인트 (`--accent`) | `#2b5bff` | `#2b5bff` (공통) |
| 포인트 호버 (`--accent-hover`) | `#5577ff` | `#5577ff` (공통) |

- **박스가 아닌 투명도**로 위계를 만든다. 다크 배경은 단일 레이어(`#0a0a0b`)이고, 텍스트는 `--text` → `--text-secondary` → `--text-muted` → `--text-faint` 네 단계 투명도로 내려가며 섹션을 구분
- **테두리는 거의 안 보이게**. `#1f1f24`는 `#0a0a0b` 배경과 대비가 약해 "선이 있긴 한데 조용한" 구분선 역할만 수행. 섹션 경계·수평 분리선·코드블록 테두리에만 사용
- **그림자는 쓰지 않는다**. 리디자인 이전 브루탈 시도가 `6px 6px 0 #2b5bff` 하드 섀도우를 썼지만 Refined Minimal 방향에선 전부 제거
- **`#2b5bff`는 딱 네 곳에만**: 링크, 포커스 아웃라인, 인라인 코드, hover 시 제목 색. 그 외에는 등장하지 않음
- 순흑(`#000`) 배경 금지, 순백(`#fff`) 본문 금지 (기존 원칙 유지)
- 코드블록 테마는 shiki 듀얼(`github-light`/`github-dark`) 유지

**기각 기록 (2026-04-05, Polished Brutalism 시도)**: `--surface: #141418`, `--border: #ffffff`(다크 반전 트릭), `--shadow-brutal-sm: 3px 3px 0 #2b5bff` 등을 시도했다가 같은 날 기각. 레퍼런스 셋(현 Gumroad·Linear·impeccable.style)과 방향 불일치, cards-on-cards 패턴 유발이 이유. 상세는 §2.5 참조.

**이전 이전 값 (리디자인 전)**: 배경 `#ffffff` / `#1a1a1a`, 본문 `#1a1a1a` / `#e8e8e8`, 테두리 `#eaeaea` / `#333`. 제네릭 AI 룩에 근접해 교체됨.

### FOUC 방지

다크모드 초기값을 React 마운트 전에 `<html>` 클래스에 적용해야 깜빡임이 없다. `index.html`에 인라인 스크립트로 `localStorage` 읽어 즉시 적용.
