이력서의 "CI 병목 해결 및 최적화" 항목을 깊이 파고든 모의 면접 기록. Gradle Daemon 동작 원리부터 Docker on Mac 아키텍처, Self-hosted Runner 보안, 측정 방법까지 한 세션 내내 한 주제를 끝까지 추궁당했다. "공식 문서를 신뢰했다", "체감 시간에 의존했다"처럼 약점이 드러난 답변들을 그대로 기록하고 어떻게 보완할지도 같이 적는다.

## 0. 이력서 요지

- **문제**: 사내망 단일 CI Runner(4 vCore / 4GB) — 동시 작업 시 빌드 대기열 발생, 전체 개발 속도 제약
- **시도 1**: 병렬 실행 → Gradle Daemon 소멸 + 컨테이너 OOM
- **제약**: Runner 서버 증설 불가
- **해결**: 사내 개발자 MacBook을 Self-hosted Runner로 활용해 CI 부하 분산
- **구성**: Linux ARM64 기반 Docker + Docker Compose로 GitHub Enterprise 엔드포인트 연동
- **보안**: VPN 환경에서만 Runner 활성화
- **결과**: 빌드 대기 시간 감소, CI 병목 해소

## 1. Gradle Daemon

### Q. Gradle Daemon이 뭔지 설명해주세요.

백그라운드에 상주하는 JVM 프로세스. 빌드가 끝나도 계속 살아 있어서 다음 빌드 시 JVM 워밍업 비용을 절감한다.

### Q. 병렬 실행 시 Daemon이 소멸한 원인을 어떻게 파악했나?

터미널 로그를 보고 확인했다. 자세한 분석은 AI에 로그를 넣어 진단을 시켰다.

### Q. AI가 분석한 결과의 원인은?

메모리 부족으로 인한 소멸. Daemon 자체가 메모리를 잡고 있는데 CI 환경에서는 Daemon이 필요하지 않다고 판단해 `--no-daemon` 옵션으로 전환했다.

### Q. `--no-daemon`의 트레이드오프는 인지하고 있었나?

Daemon을 안 쓰면 재빌드 시 JVM 프로세스를 재사용하지 못해 빌드 속도가 느려진다. 다만 너무 적은 자원으로 멀티모듈을 병렬 빌드하기 어려워 `--no-daemon` + 병렬 처리 자체를 포기했다.

### 보완 포인트

- Daemon 메모리 점유의 본질은 **JVM heap + metaspace + 빌드 캐시**. `org.gradle.jvmargs=-Xmx`로 heap을 줄이는 옵션을 먼저 시도해봤어야 한다는 답변을 미리 준비
- "AI가 분석했다"보다는 "로그에서 OOM Killer 흔적/`Daemon disappeared` 메시지를 확인하고 AI로 가설을 검증했다"가 더 안전한 표현

## 2. Mac에서 Linux ARM64 Docker가 도는 원리

### Q. Mac에서 Linux ARM64 Docker 컨테이너를 띄운다는 게 구체적으로 어떤 의미?

Docker에 Ubuntu 이미지를 올리고 그 안에서 GitHub Runner 등록 스크립트를 실행해, 각 MacBook이 Runner 서버 역할을 한다.

### Q. Mac은 Darwin 커널 기반인데 그 위에서 Linux 컨테이너가 돌아가는 구조는?

Docker가 자동으로 리눅스 VM을 띄워주는 걸로 안다. 자세히는 모르고 Docker Desktop 기본 설정을 그대로 쓰고 있다.

### Q. 그 부분을 더 깊이 알고 싶다면 어떻게 알아볼 것?

Docker 공식 문서에서 Mac 아키텍처 다이어그램을 먼저 본다.

### 보완 포인트

면접에서 모를 때 정직하게 모른다고 한 건 좋지만, 한 줄 정도는 더 붙일 수 있어야 한다:

> "Docker Desktop이 LinuxKit 기반의 경량 VM을 띄우고, 그 안에서 컨테이너가 실행됩니다. Apple Silicon에서는 Virtualization.framework를 쓰는 걸로 알고 있고, ARM64 이미지를 쓰면 에뮬레이션 없이 네이티브로 돕니다."

이 정도는 알고 있어야 "Linux ARM64로 구성했다"는 이력서 문장이 단순한 카피가 아니라는 게 증명된다.

## 3. Self-hosted Runner 보안

### Q. "보안상 안전하다"고 판단한 근거는?

회사 VPN 환경에서만 GitHub Enterprise에 접근 가능. VPN을 안 쓰면 Runner도 작동하지 않는다. 즉 회사 사람이 아니면 접근 자체가 불가능하다고 판단했다.

### Q. VPN에 접속한 사람이라면 누구든 Runner로 등록할 수 있나? Runner 등록의 인증/권한 통제는?

공통 Runner가 아니라 **레포지토리 Runner**다. 누구나 등록할 수 있는 구조지만, 해당 레포지토리의 권한을 가진 사람만 등록이 가능하다.

### Q. 권한자가 본인 MacBook에 Runner를 등록하면 그 위에서 Job이 실행되는데, 환경변수로 주입되는 시크릿이나 빌드 산출물이 MacBook에 남을 위험은?

Docker로 실행하고, Runner 등록 시 자체적으로 모든 정보를 지우고 초기화하는 옵션이 있다. 이 옵션을 통해 기존 설정과 충돌 없이 사용한다.

### Q. 그 옵션이 구체적으로 어떤 옵션인지 기억하나?

지금은 자세히 기억나지 않는다.

### Q. 설정을 어떻게 찾아서 적용했나?

공식 문서를 찾아보거나 AI에 물어본 후 검증했다.

### Q. 시크릿이 실제로 MacBook에 안 남는다는 건 어떻게 검증?

Docker 컨테이너에 직접 들어가 환경변수가 남아있는지 확인했다.

### Q. Job이 끝난 후 컨테이너가 살아있어서 들어갈 수 있었다는 건가? 컨테이너 라이프사이클은 어떻게 구성?

Runner의 특정 설정으로 프로세스가 종료되면 Docker도 같이 종료되는데, Compose에서 종료되지 않도록 `restart: always`로 설정해놔서 다시 뜬다.

### Q. 컨테이너가 재시작되는 구조라면, 새로 뜨는 시점에 이전 Job의 데이터가 컨테이너 볼륨이나 Docker 레이어에 남아있을 가능성은 검토했나?

GitHub 공식이 권장하는 구조이고, 옵션을 켜면 초기화된다고 나와 있어서 따로 확인하지 않았다.

### 보완 포인트

이 섹션이 가장 약했다. **"공식 문서를 신뢰했다"는 답변은 면접에서 매우 위험**하다. 보완 방향:

- "마운트한 워크스페이스 볼륨을 Job 종료 시 비우는 hook을 별도로 구성하고, 검증은 Job 직후 환경변수/파일/이미지 캐시 3곳을 모두 확인했어야 한다"
- 기본적으로 GitHub Actions Runner는 Job 종료 후 워크스페이스를 자동 정리하지만, **컨테이너 레이어 캐시·도커 볼륨·`/tmp` 등은 별개**라는 점을 알고 답해야 한다
- 답변 라인: "공식 권장 구조를 따랐고, 추가로 환경변수가 컨테이너 종료 후 새 인스턴스에 남지 않는지 직접 컨테이너에 진입해 확인했습니다. 다만 Docker 레이어 캐시까지는 검증하지 못했고, 다음에 한다면 `--rm` 기반 일회성 컨테이너 + 외부 볼륨 분리로 가져갈 것 같습니다."

## 4. MacBook Runner의 근본 한계와 Fallback

### Q. 개발자가 MacBook을 끄거나 VPN을 끊으면 Runner가 내려간다. 그 상황에서 Job이 할당되면?

CI가 종료된다. 다만 큰 문제는 없는데, CI 실패가 PR에 표시되고 PR은 CI 실패 시 머지가 막히기 때문이다. 사용자가 직접 재실행하면 다른 Runner에서 다시 시작된다.

### Q. 모든 MacBook Runner가 오프라인이라면 Job이 계속 대기 상태에 머무는데, fallback은?

공통 서버 자산이 있다. 종료되지 않는 서버라 그쪽에서 진행한다.

### Q. 그러면 처음의 4 vCore / 4GB 단일 노드 서버가 fallback으로 병존하는 구조네요. GitHub Actions에서 MacBook → 서버 우선순위는 어떻게 제어?

GitHub의 기본 동작을 그대로 사용했다. 내부 스케줄링 로직은 모르고 알아서 분배되니 신경쓰지 않았다.

### Q. 지금 다시 이 문제를 푼다면 다른 접근 방식을 고려해봤을 것 같은데?

굳이 더 해결할 문제는 아니라고 본다. 사용자가 일을 할 때 Runner가 켜져 있으면 되고, 실패해도 수동 재실행이 가능하며, 일하는 사람이 없을 때 Runner가 살아있을 필요도 없다. 우리 서비스의 비즈니스 로직 구현이 더 중요하다.

### 보완 포인트

마지막 답변은 의도는 좋으나 면접에서는 **"개선 의지 없음"**으로 들릴 수 있다. 더 안전한 라인:

> "현재 구조는 비용 대비 효과가 좋아서 더 투자하지 않았습니다. 다만 더 키워야 한다면 Runner Group으로 우선순위를 명시하고, 라벨 기반 라우팅으로 무거운 Job은 서버 Runner, 가벼운 Job은 MacBook으로 분리하는 식이 자연스러울 것 같습니다."

GitHub Actions의 Runner 우선순위는 **라벨 기반**이고, `runs-on`에 라벨을 명시하면 매칭되는 Runner 풀에서 가져간다는 사실 정도는 알고 있어야 한다.

## 5. 측정과 결과

### Q. "CI가 더 이상 병목이 아니다"의 측정은? 구체적인 수치는?

PR 횟수와 일감 몰림에 따라 다르겠지만, 10~20분 대기하던 게 1~2분이면 해결되었다.

### Q. 어떻게 측정? GitHub Actions 로그? 별도 모니터링?

GitHub Actions의 총 시간을 적용 전과 후로 비교했다.

### Q. GitHub Actions가 보여주는 건 실제 빌드 실행 시간이고, 대기 시간은 Job이 큐에 들어간 시점부터 Runner가 집어가는 시점까지인데, 그 대기 시간 자체는 어떻게 측정?

자세하게 따로 시간을 측정하지는 않았다. 작업량에 따라 천차만별이라 변경 후 체감 시간에 의존했다.

### 보완 포인트

이 답변도 약점이 그대로 드러났다. **"체감 시간"은 면접에서 절대 쓰면 안 되는 단어**다. 보완:

- GitHub Actions의 Job 페이지에는 `Queued at` / `Started at` / `Completed at` 타임스탬프가 있고, API(`GET /repos/{owner}/{repo}/actions/runs/{run_id}/jobs`)로도 가져올 수 있다. 이 차이가 큐 대기 시간이다.
- 답변 라인: "GitHub Actions API에서 `started_at - created_at`을 큐 대기 시간으로 정의하고, 도입 전후 한 주씩 평균을 비교했어야 합니다. 당시에는 이 부분을 정량화하지 못했고, 체감 기준으로 10~20분 → 1~2분 정도라고만 설명드릴 수 있습니다."

"못 했다"고 명확히 인정한 뒤 "한다면 X"를 붙이는 패턴.

## 전체 평가

### 강점

- **트레이드오프 인지**: `--no-daemon`의 단점(JVM 워밍업 비용 손실)을 정확히 알고 있었음
- **솔직한 모름 인정**: Mac Docker 내부 구조를 모를 때 추측 대신 "모른다 + 어디서 찾을 것"으로 답한 건 좋은 패턴
- **Runner 등록 권한 모델**: 레포지토리 Runner의 권한 통제가 무엇에 의존하는지 정확히 답변
- **현실적 우선순위 판단**: 비즈니스 로직 우선이라는 판단 자체는 시니어 답변

### 약점 패턴

- **"공식 문서를 신뢰했다" 1번** — 시크릿이 컨테이너에 안 남는지에 대한 검증이 얕았음. 공식 문서를 신뢰하는 것과 직접 검증하는 것은 다른 차원
- **"AI에 분석 시켰다" 1번** — Gradle Daemon 소멸 원인 파악. AI를 도구로 쓴 건 좋지만 "내가 가설을 세우고 AI로 검증했다"로 톤을 바꿔야 한다
- **"체감 시간에 의존했다" 1번** — 정량 지표 없는 결과 보고. 면접에서 가장 치명적
- **"기본 동작을 그대로 썼다" 2번** — 우선순위 스케줄링과 컨테이너 라이프사이클. "그대로 썼다"는 "왜 그렇게 동작하는지 모른다"로 들린다
- **"굳이 해결할 필요 없다" 1번** — 의도는 맞지만 면접에서는 "다만 더 키운다면 X"를 반드시 붙여야 한다

### 다음 모의 면접 전 정리할 것

1. **Docker on Mac 아키텍처 한 줄 답** — LinuxKit VM + Virtualization.framework, ARM64 이미지는 네이티브 실행
2. **GitHub Actions 큐 대기 시간 측정 방법** — `started_at - created_at`, API로 정량화 가능
3. **Self-hosted Runner 보안 체크리스트** — 워크스페이스 정리 / 컨테이너 레이어 / 볼륨 / 시크릿 마운트 4축
4. **Runner 라벨 기반 라우팅** — `runs-on` 라벨로 Runner 풀 분리, 무거운 Job/가벼운 Job 분배 전략
5. **"못 했지만 한다면 X" 답변 템플릿** — 미측정·미검증 항목별로 1줄씩 미리 준비
