## 1. Cloud Map 선택 이유

### Q. Cloud Map이 뭔가?

같은 VPC 안에서 내부 도메인을 고정해주는 서비스. K8s FQDN과 비슷한 개념이고, 외부 IP가 바뀌어도 내부 도메인은 그대로 유지된다.

### Q. 왜 Cloud Map인가? 다른 선택지는?

- **직접 IP 통신**: ECS 태스크가 자주 죽고 살아나며 NIC가 매번 바뀌어 고정 IP 확보 어려움. EIP는 과금 부담
- **Internal ALB**: ALB는 가격 부담
- **Cloud Map**: 가격 부담 적고 ECS 태스크 등록 자동화

### Q. 태스크가 죽고 새로 뜨는 사이 시간 동안 다른 서비스가 죽은 엔드포인트로 요청을 보내면?

애플리케이션에서 에러가 난다. 이건 retry로 해결한다.

### 보완 포인트

답변은 비용 기반으로 정리되어 좋았다. 한 줄 강화:

> "Cloud Map은 ECS 서비스 디스커버리와 통합되어 태스크 IP 변화를 자동 등록/해제해주고, Route 53 Private Hosted Zone에 A/SRV 레코드를 만들어줍니다. Internal ALB는 L7 라우팅과 헬스체크가 강점이지만 시간당 고정 비용이 있고, 우리 트래픽 규모에서는 Cloud Map의 DNS 기반이 충분했습니다."

## 2. Retry 정책

### Q. Retry는 어느 레이어, 어떤 방식?

레이어드 아키텍처의 서비스 레이어. **애노테이션 기반 라이브러리**(Spring Retry)로 3회 재시도, 0.5초 간격. Cloud Map에서 실패해도 다음 요청은 다른 태스크로 갈 수 있어 간격을 짧게 했다. 3번 모두 실패하면 서버 전체 장애로 판단.

### Q. 3회 재시도 후에는 어떻게?

MSA가 아니라 FE 직접 호출 구조라 별도 처리는 적었다. 최종적으로 5xx가 나면 "서버 연동 실패" 메시지를 사용자에게 노출.

### 보완 포인트

retry 정책 자체는 합리적이지만 다음을 추가해야 시니어 답변:

- **Idempotency 인지**: GET/PUT/DELETE는 안전하지만 POST는 retry 시 중복 생성 위험. 멱등성 키 또는 메서드별 분기 필요
- **Exponential backoff + Jitter**: 고정 0.5초보다 점증 + 랜덤 지터가 thundering herd를 피함
- **Circuit Breaker와의 결합**: retry는 일시적 실패에, 서킷 브레이커는 지속적 실패에 대응

답변 라인:

> "Spring Retry로 3회 / 0.5초 간격을 적용했고, GET 메서드에만 자동 적용했습니다. POST 같은 비멱등 호출은 멱등성 키와 함께 별도 정책으로 갔습니다. 트래픽이 더 커지면 exponential backoff + jitter로 전환하고, 지속적 실패는 retry보다 서킷 브레이커가 적합합니다."

## 3. 헬스체크 공백 시간

### Q. 운영 중 태스크가 갑자기 죽으면 ALB가 unhealthy로 감지하기까지?

ALB 헬스체크 10초 간격, 3회 실패 기준. 최악 30초 동안 502가 발생할 수 있다.

### Q. 30초를 허용 가능하다고 판단한 근거?

사용자가 많지 않은 서비스, 트래픽이 적어 큰 문제가 안 됨. 프론트 재시도가 다른 서버로 라우팅해줌.

### Q. 토스급 결제 서비스라면 그 30초 허용을 유지할 건가?

5초 간격 / 2회로 줄여 최악 10초로 단축. 1회는 오탐 위험이라 최소 2회 유지.

### Q. 간격/횟수 조정 외에 공백 자체를 줄이는 방법은?

- **Container health check** 추가로 ALB보다 먼저 감지
- **서킷 브레이커**로 코드 레벨에서 빠르게 차단

### 보완 포인트

공백 시간 단축의 본질은 **빠른 unhealthy 감지 + 빠른 트래픽 회수**이다. 추가로 가능한 방법:

- **ALB Connection Draining timeout** 단축으로 이미 빠진 인스턴스의 연결 잔여 시간 줄이기
- **ECS Task의 graceful shutdown 시그널 처리**로 SIGTERM 받자마자 ALB에 deregistration 신호 + in-flight 요청 마무리
- **Outlier detection** (Envoy/Istio 환경에서는 클라이언트 사이드에서 비정상 응답 비율로 자동 격리)

답변 라인:

> "ALB 헬스체크 공백을 직접 줄이는 것 외에, ECS Task가 SIGTERM을 받으면 즉시 ALB에서 deregister하고 in-flight를 정리하는 graceful shutdown 훅이 더 근본적입니다. 컨테이너가 스스로 '나 곧 죽음'을 알려줄 수 있으면 ALB의 수동 감지 대기 자체가 사라집니다."

## 4. 서킷 브레이커 위치 — 잘못된 답변

### Q. 서킷 브레이커를 어디에 적용?

답변: "프론트가 백엔드를 호출하는 곳."

### Q. 서킷 브레이커 동작?

CLOSED → OPEN → HALF-OPEN 3상태. CLOSED일 때 정상 호출, 실패율이 임계치 넘으면 OPEN으로 fallback 반환, 일정 시간 후 HALF-OPEN으로 일부 요청 통과시켜 복구 여부 확인.

### Q. FE가 직접 백엔드를 호출하는 구조에서 FE에 서킷 브레이커를 적용하는 게 현실적인가?

답변: "프론트 재시도는 있는데 서킷 브레이커는 모르겠다."

### 보완 포인트

이 섹션이 가장 약했다. **서킷 브레이커는 서버 사이드에서 다른 서비스를 호출할 때 적용하는 패턴**이다. FE에 적용하는 건 일반적이지 않고, MSA라면 BFF나 API Gateway에 두는 게 정석이다.

이 서비스 구조에서는 사실상 서킷 브레이커가 적합한 위치가 없다. 솔직한 답변:

> "현재 구조는 FE → ALB → Backend로 백엔드가 다른 서비스를 호출하지 않는 단일 서비스라 서킷 브레이커가 들어갈 자리가 없습니다. MSA로 전환되면 백엔드 간 호출 지점에 Resilience4j 같은 라이브러리로 적용하는 게 정석입니다."

"FE에 적용한다"는 답이 아니라 "지금 구조에는 들어갈 자리가 없다"가 정답이었다.

## 5. 2대 동시 다운 시나리오

### Q. 2대 서버가 동시에 죽으면?

프론트에서 서버 호출이 안 된다.

### Q. 그 상황 대비책은?

ECS가 자동으로 새 태스크를 띄워준다.

### Q. 5분 걸린다고 했는데, 2대가 동시에 죽으면 5분 동안 완전 중단 아닌가?

- 안정성을 더 원한다면 2대 → 여러 대로 증설
- ECS Task type을 EC2 → Fargate로 전환해 기동 시간 단축
- K8s 마이그레이션도 옵션

### Q. 현재 EC2를 선택한 이유?

당시 학습 시간이 부족해서 알고 있는 자원으로 구축. Fargate가 소규모에서 더 저렴하고 빠른 배포 가능.

### 보완 포인트

"5분 동안 멈춰도 괜찮다"는 답은 **트래픽이 없는 사이드 프로젝트에서만 유효한 절충**이다. 본질은 **MTTR(Mean Time To Recovery) vs 비용**의 트레이드오프. 답변 라인:

> "2대 동시 다운은 RTO 5분이라는 비공식적 가용성 합의를 깔고 있고, 트래픽이 거의 없는 시간대 + 사용자 영향도가 낮다는 전제로 허용했습니다. 가용성 SLA를 99.9% 이상으로 올리려면 (1) Fargate로 기동 시간 단축, (2) 더 많은 replica + AZ 분산, (3) Blue/Green 배포로 새 버전이 활성화될 때까지 기존 태스크 유지 같은 옵션이 차례로 들어옵니다."

## 6. EBS → ECS 전환 결정

### Q. Elastic Beanstalk에서 ECS로 전환한 핵심 이유?

- **회사 전반 인프라 공통화**
- **Beanstalk의 세부 설정 미흡**: 배치 Docker + 서비스 Docker 등 여러 컨테이너가 같이 떠야 하는데 당시 Beanstalk는 멀티 컨테이너 지원이 부족했다

### 보완 포인트

이 답변은 짧지만 정확하다. 한 줄 보강:

> "Beanstalk의 multi-container Docker 플랫폼이 deprecated 방향으로 가고 있었고, ECS Task Definition은 컨테이너 단위 리소스 할당, 사이드카 패턴, 네트워크 모드 선택 등 세밀한 제어가 가능합니다. Cloud Map / Service Discovery / Capacity Provider 같은 ECS 생태계와의 통합도 결정 요인이었습니다."

## 전체 평가

### 강점

- **비용 기반 의사결정** — Cloud Map vs Internal ALB, EC2 vs Fargate에서 트래픽/비용을 명확히 비교
- **헬스체크 공백 시간을 정량으로 인지** — 30초라는 숫자를 정확히 계산해본 흔적
- **트레이드오프 인식** — "토스급이라면" 가정 질문에 즉시 5초 / 2회로 보정
- **솔직한 모름 인정** — Container Health Check, 서킷 브레이커 적용 위치 등 모르는 부분을 회피하지 않음

### 약점 패턴

- **서킷 브레이커 적용 위치를 잘못 답변** — FE에 적용한다는 답은 패턴 자체의 정의를 흔든다
- **2대 동시 다운에 대한 즉답이 약함** — "ECS가 새로 띄워준다"로 끝낸 첫 답변은 MTTR/SLA 관점이 빠짐
- **Retry 정책에 멱등성 언급 없음** — POST 재시도 위험을 의식하지 않은 게 약점
- **"공부할 시간이 부족했다"** — Fargate 미선택 이유로 쓰면 면접관이 "그러면 지금이라면?"으로 즉시 들어옴. 답변 깔고 가야 함

### 다음 모의 면접 전 정리할 것

1. **Cloud Map vs Internal ALB vs Service Connect** 한 줄 비교
2. **Spring Retry + 멱등성** 답변 라인 — "GET에만 적용, POST는 멱등성 키와 함께"
3. **ALB graceful shutdown** — SIGTERM → deregister → in-flight 정리 흐름
4. **서킷 브레이커는 서버↔서버에만** — FE에 적용하는 패턴이 아님을 명시
5. **MTTR vs SLA 트레이드오프 한 문장** — "RTO 5분은 ~의 절충, 99.9% SLA가 필요하면 ~"
6. **EC2 vs Fargate 의사결정 매트릭스** — 비용/기동시간/관리 부담/스케일 패턴
