## 0. 이력서 요지

- **문제**: 5~6천만 건 데이터 테이블에서 그래프 1개 조회에 165초
- **시도 1**: 인덱스 → 약 2배 빨라짐 (85초). 여전히 사용 불가
- **시도 2**: 일별/월별 집계 테이블 도입 → 0.02초로 단축
- **시도 3**: 캐싱 (현재 날짜 제외) → 0.0005초까지 단축
- **구조 개선**: 조회 기간에 따라 원본/일별/월별 테이블을 선택해야 하는 분기 → 팩토리 패턴 + 전략 패턴으로 결합도 분리

## 1. 집계 테이블 DDL과 인덱스

### Q. 일/월/원본 집계 테이블의 핵심 컬럼을 DDL 수준으로?

세 테이블 모두 동일한 컬럼: `관제점`, `날짜`, `값`. PK는 auto-increment.

### Q. (날짜, 관제점) 순서로 잡았다고 했는데 왜?

처음 답변: "날짜만으로 조회할 일이 있을 것 같아서 날짜를 먼저 잡았다. 다만 운영해보니 항상 두 조건이 같이 들어왔다."

→ 면접관이 "그러면 (관제점, 날짜)가 더 나았을 텐데?"라고 짚자 정정:

**정정 답변**: 사실 (관제점, 날짜) 순서로 잡았다. 관제점으로 먼저 카디널리티가 높은 쪽을 줄이고, 그다음 날짜 범위로 좁히는 게 row를 가장 적게 읽는 구조다.

### Q. 카디널리티 비교 근거는?

관제점 100개, 관제점당 데이터 100개 가정 시:
- (날짜, 관제점) 순서: 날짜 범위로 100×100 안에서 일단 다 훑은 뒤 관제점 필터링
- (관제점, 날짜) 순서: 관제점 단일 값으로 먼저 100개로 좁히고, 그 안에서 날짜 범위 스캔 → 훨씬 적은 row

### Q. 실제 운영 데이터의 카디널리티는?

- 관제점: 1000개 이상
- 데이터 적재: 5분당 1건
- 원본 테이블: 5~6천만 건
- 일별 집계 테이블: 관제점 1000개 × 일수 ≈ 수십만 건 수준

### 보완 포인트

**(날짜, 관제점)이라고 처음 답한 게 면접관에게 즉시 잡혔다.** 이건 이력서에 적은 내용을 머릿속에 정확히 박지 않은 상태에서 면접에 들어간 신호다. 인덱스 컬럼 순서는 면접에서 거의 100% 물어보는 질문이라 다음 면접 전까지 답변 라인을 외워야 한다:

> "(관제점, 날짜) 순서입니다. 관제점은 카디널리티가 1000 이상이라 단일 값 매칭으로 후보를 빠르게 좁히고, 그다음 날짜를 범위 스캔으로 처리하는 게 B-Tree 인덱스의 leftmost prefix 규칙과도 맞습니다. 만약 날짜 단일 조회가 필요하면 그건 별도 인덱스로 풀어야지, 복합 인덱스 순서를 바꾸는 건 잘못된 트레이드오프입니다."

## 2. 집계의 의미 — 무엇을 저장했나

### Q. 5분 단위 원본을 일별로 집계할 때 "값" 컬럼에 무엇을 저장?

합계.

### Q. 그래프에 보여주는 게 합계인가?

거의 대부분의 상황에서 합계, 그리고 합계 기반의 추가 계산이 이뤄진다.

### Q. 합계가 아닌 경우는?

먼저 합계를 가져오고 애플리케이션 내부 로직에서 평균이나 계산식을 적용한다.

### Q. "3월 한 달간 일별 평균"을 사용자가 요청하면?

서비스가 건물 에너지 측정이라 일 단위 평균은 사실 거의 안 찍는다. 한 달/연 단위, 또는 "하루에 얼마 썼는가"가 주된 사용 패턴. 만약 굳이 한다면 **일 합계 ÷ 288**(5분 단위 행 개수)을 적용하면 된다.

### 보완 포인트

이 답변은 즉석 계산으로 이어 붙여서 위태로웠다. **집계 테이블에 합계만 저장하는 설계의 본질적 한계**는 다음과 같다:

- 평균: 합계 ÷ count(원본 row 수)로 복원 가능 (count는 288 같은 상수일 때만)
- 최솟값/최댓값: **복원 불가**. 일 합계만으로는 그 날의 분 단위 최고/최저값을 알 수 없음
- p95/p99 같은 분위수: **복원 불가**

답변 라인:

> "집계 테이블의 '값' 컬럼은 합계만 저장합니다. 평균은 5분 단위 행 개수가 일정해서 합계에서 복원 가능하지만, **최댓값/최솟값/분위수처럼 복원 불가능한 통계가 필요하면 집계 테이블에 별도 컬럼(min, max)을 같이 저장**해야 합니다. 현재 서비스의 사용 패턴은 합계와 합계 기반 평균이 전부라 이렇게 간단히 두고 있습니다."

## 3. 기간 라우팅 규칙

### Q. 어떤 기간에 어떤 테이블을 쓰나?

프론트가 보내주는 단위 기준:
- 분/시간 단위 → 원본 테이블 (최대 3일)
- 일 단위 → 일별 집계 테이블 (최대 5년)
- 월/년 단위 → 월별 집계 테이블 (제한 없음)

### Q. "오늘 하루"든 "지난 1년치"든 일 단위는 모두 일별 집계?

맞다. 단위 자체로 분기.

## 4. 캐싱 — 키, TTL, 분산

### Q. 캐시 키 설계는?

처음: "건물 ID + 조회 타입 + 시작날짜 + 종료날짜 + 관제점 ID 리스트"

→ 면접관이 "리스트 순서가 달라지면 같은 데이터가 다른 키가 되는 문제는?"이라고 짚자:

**정정**: 사실 관제점 ID가 아니라 **관제점 타입**이었다.

### Q. 캐시 기술과 TTL은?

- 기술: **Caffeine 로컬 캐시 (in-memory)**
- TTL: 하루
- 메모리 제약: 최대 메모리 사용량 제한 설정 (정확한 값은 기억 안 남)

### Q. 여러 인스턴스로 배포되는 구조였는데 로컬 캐시 정합성은?

여러 인스턴스로 배포되는 게 맞고, 로컬 캐시라 인스턴스마다 따로 캐싱된다는 점은 인지하고 있었다. 두 가지 이유로 그대로 갔다:

1. 데이터 자체가 작아서 중복 캐싱이 큰 부담이 아니었다
2. Redis를 띄우는 비용을 회사에서 지원해주지 않았다

### Q. 배치로 집계 데이터를 다시 계산해야 하면 각 인스턴스의 로컬 캐시는?

`condition`에 오늘 날짜가 포함되면 캐싱 자체가 안 되도록 막아두긴 했다. 만약 과거 데이터를 수정해야 하는 상황이 오면 해당 캐시를 무효화해야 한다.

### Q. 여러 인스턴스의 로컬 캐시를 한 번에 무효화하려면?

이런 케이스가 생기면 Redis를 도입해 공통 캐시로 가져가는 게 맞다. 현재는 그럴 필요가 없어서 안 썼고, 필요해지면 그때 마이그레이션할 것이다.

### 보완 포인트

이 섹션의 답변은 전반적으로 **"필요해지면 Redis"**라는 자세를 일관되게 유지한 게 좋았다. 다만 다음 답변 라인을 미리 준비하면 더 강하다:

> "현재는 로컬 캐시 + 오늘 날짜 condition 차단으로 충분합니다. 다만 두 시나리오에서 분산 캐시 도입을 고려할 겁니다. (1) 과거 집계가 재계산되는 비즈니스 요구가 생길 때, (2) 캐시 히트율이 낮아져서 인스턴스마다 같은 데이터를 중복 캐싱하는 비용이 메모리 비용보다 커질 때. 그 전까지는 Redis 인프라/운영 비용이 더 비쌉니다."

"필요해지면 한다"가 아니라 "이 조건이 깨지면 한다"로 전환 기준을 명시하는 게 시니어다운 답변.

## 5. @Cacheable + condition

### Q. Spring `@Cacheable`을 썼다고 했는데 SpEL key/condition은?

```kotlin
@Cacheable(
  value = "...",
  key = "#buildId + '_' + #startDate + '_' + #endDate + '_' + #itemType",
  condition = "#endDate.before(T(java.time.LocalDate).now())"
)
```

`condition`에서 `endDate.before(now)`로 종료일이 오늘보다 이전인 경우만 캐싱.

### Q. 종료일이 정확히 오늘이면?

`before`는 같은 날짜에서 false를 반환하므로 캐싱이 안 된다.

### 보완 포인트

답변은 정확하지만 한 가지 함정이 있다. **`@Cacheable`은 메서드 진입 시점에 condition을 평가**하므로, 사용자가 "오늘 ~ 어제"처럼 잘못된 범위를 보내면 condition 분기에서 의도와 다르게 동작할 수 있다. 또한 `LocalDate.now()`는 서버 타임존에 의존한다. 답변 라인:

> "condition은 메서드 진입 시 평가되므로 종료일이 오늘이면 캐싱 차단되고, 그 이전이면 캐싱됩니다. `LocalDate.now()`는 서버 타임존에 의존하므로 멀티 리전 배포가 되면 ZoneId를 명시해야 합니다."

## 6. 팩토리 패턴 — 자기 부정 후 재긍정

### Q. 팩토리 클래스 구조를 코드 수준으로?

팩토리 안에 분/일/월 서비스 3개를 의존성으로 주입받고, switch-case로 기간 타입에 따라 알맞은 서비스를 반환한다.

답변 도중 자기 의심: "그때는 팩토리 메서드 패턴이라고 생각했는데 지금 보니 팩토리 패턴이 아닌 것 같다."

→ 면접관이 "그러면 어떤 패턴에 가깝나?"라고 묻자:

**다시 정정**: 팩토리 메서드 패턴이 아니라 (단순) 팩토리 패턴을 구현한 것 같다.

### Q. 분/일/월 서비스가 공통 인터페이스를 구현하나?

맞다. **전략 패턴**으로 같은 인터페이스를 구현하고, 팩토리가 그 인터페이스를 반환한다. 인터페이스 메서드는 "관제점과 날짜를 받아 관제값 리스트를 반환".

### Q. 구현체 내부에서 직접 쿼리? 아니면 별도 Repository?

별도 Repository. 세 구현체가 각각 다른 Repository를 주입받음. 구현체 간 공통 로직은 단순 DB 조회뿐이라 추가로 묶을 게 없었다.

### Q. 호출 흐름 (컨트롤러 → 어디 → 어디)?

1. Controller가 요청 받음
2. AppService로 위임
3. AppService가 날짜 타입을 팩토리에 전달
4. 팩토리가 알맞은 구현체를 반환
5. 구현체가 자신의 Repository로 데이터 조회

### Q. 캐싱 로직은 어느 레이어?

AppService 레이어. 같은 요청이라면 굳이 구현체까지 내려가지 않아도 되니 위에서 잘랐다.

### Q. 캐시 히트 시 팩토리 호출 자체를 건너뛰나?

맞다. AppService 메서드에 `@Cacheable`이 붙어 있어서 캐시 히트면 메서드 본문(팩토리 호출 포함)이 실행되지 않는다.

### 보완 포인트

이력서에 쓴 패턴 이름을 자기 입으로 부정했다가 다시 긍정한 게 가장 약한 부분이다. 면접관 입장에서는 "자기 코드의 패턴 이름조차 확신이 없다"로 들린다.

**팩토리 메서드 패턴 vs 단순 팩토리(Simple Factory) vs 추상 팩토리** 세 개를 다음 면접 전 한 줄씩 정리:

| 패턴 | 핵심 |
|------|------|
| Simple Factory | 일반 클래스가 정적/인스턴스 메서드로 분기해서 객체 반환. GoF 23패턴은 아님 |
| Factory Method | 추상 메서드를 서브클래스가 오버라이드해서 반환 객체를 결정 |
| Abstract Factory | 서로 관련된 객체군(family)을 만드는 팩토리들의 팩토리 |

답변 라인:

> "엄밀히는 GoF의 Factory Method가 아니라 Simple Factory에 가깝습니다. 팩토리 클래스 1개가 switch-case로 전략 객체를 반환하는 구조이고, 반환 타입은 전략 인터페이스입니다. 즉 **Simple Factory + Strategy의 조합**입니다."

이 한 문장이면 패턴 정의에 대한 신뢰가 즉시 회복된다.

## 전체 평가

### 강점

- **합계 기반 집계 + 애플리케이션 후처리** 분리 인지
- **로컬 캐시 vs Redis 트레이드오프** — 비용/필요성 기준으로 명확히 결정한 자세
- **`@Cacheable` + condition + SpEL** 실무 적용 경험
- **팩토리 + 전략 패턴 조합** — 구조 자체는 정확히 만들었음

### 약점 패턴

- **인덱스 컬럼 순서 잘못 답변 후 정정** — 면접의 단골 질문에서 즉답이 틀린 건 가장 치명적
- **캐시 키 구성 잘못 답변 후 정정** — "관제점 ID 리스트"라고 했다가 "관제점 타입"으로 정정. 본인 코드의 키 구성을 정확히 기억하지 못함
- **패턴 이름 자기 부정 후 재긍정** — 이력서에 쓴 패턴 이름을 면접에서 흔들면 신뢰가 즉시 무너짐
- **"필요해지면 한다"** — 좋은 자세지만 "이 조건이 깨지면 한다"로 임계값을 명시해야 시니어로 들림
- **즉석 계산 답변** — "288로 나누면 됩니다" 같은 답변은 위험. 사전에 외워두지 않으면 실수가 나옴
