## 0. 이력서 요지

- **문제**: 규모가 큰 프로젝트에서 단일 서버의 비대함 + 배포 단위 과대
- **전환 단계**: DDD(이벤트 스토밍) → 모노레포 멀티 모듈 → 헥사고날 → MSA
- **공통 모듈**: 인증/인가, JPA 설정, 공통 유틸
- **동기 vs 비동기**: 요청-응답 책임이 분명한 곳만 동기, 대량/지연 허용은 비동기
- **결과**: 서비스별 독립 배포 + 독립 장애 범위 확보

## 1. 바운디드 컨텍스트 분리 기준

### Q. 이벤트 스토밍으로 도출한 BC는?

총 3개:
- **Kubernetes**: workload, namespace, storage
- **Portal**: 프로젝트, 권한
- **Member**: 인증

### Q. Portal의 "권한"과 Member의 "인증/인가"가 밀접한데 분리 기준은?

- **인증**: Keycloak 공통 처리
- **인가**: 공통 모듈 처리
- **Portal의 권한**: 프로젝트/조직별 권한. 한 명의 멤버가 다른 조직에 가면 다른 권한을 가짐 → 권한 주체는 멤버가 아니라 "조직 안의 멤버"라서 Portal에 위치
- **Kube vs Portal**: "쿠버네티스 API를 다루는가" vs "우리 서비스 도메인을 다루는가"로 경계

### 보완 포인트

답변은 명확했다. 한 줄 강화:

> "BC 분리의 핵심은 '변화의 이유'입니다. 멤버는 사람 자체의 식별과 자격 증명이 변화의 이유이고, Portal의 권한은 조직 구조나 프로젝트 정책이 변화의 이유입니다. 둘이 다른 속도로 바뀌기 때문에 같은 BC에 두면 한쪽의 변경이 다른 쪽을 끌어가게 됩니다."

SRP의 변종인 "변화의 이유가 같은가"로 프레이밍하면 DDD 질문에 강해진다.

## 2. 권한 동기 호출 — 정합성 vs 가용성

### Q. Kube 서비스가 Portal의 권한을 어떻게 확인?

Spring Security의 `hasPermission`을 전략 패턴으로 확장. Kube 요청이 들어오면 Portal에 **동기 호출**로 승인/거부를 판단.

### Q. Portal 장애 시 Kube가 블로킹되는데?

- Portal Pod 2개 이상으로 띄워 가용성 확보
- 완전히 다 죽으면 사용자에게 적절한 에러 메시지

### Q. 타임아웃/서킷 브레이커/캐싱 적용은?

- **서킷 브레이커**: 적용, fallback으로 에러 반환
- **권한 캐싱**: 의도적으로 배제. 권한은 민감한 정보라 캐시와 실제 값의 불일치 시 "방금 삭제 권한을 뺐는데 캐시 때문에 삭제 가능"한 상황이 치명적

### Q. 결국 정합성 택하고 가용성 포기한 건데, 읽기 vs 쓰기 권한을 분리해 전략을 다르게 가져가는 건 검토했나?

안 했다.

### Q. 다시 설계한다면?

정합성을 우선하는 게 맞다고 생각. 권한이 Portal 단일 주체가 알고 있는데 Kube에도 정의하면 중복/불일치가 생긴다.

### 보완 포인트

이 섹션이 이번 세션의 가장 중요한 부분이다. 면접관이 정확히 짚은 포인트는 **"읽기/쓰기 위험도 분리"**. 답변 라인:

> "지금 다시 설계한다면 권한을 위험도별로 분리합니다. 읽기 권한은 TTL 짧은 캐시 + Portal 장애 시 stale 허용, 쓰기 권한(삭제/수정)은 항상 Portal 동기 확인, 그리고 권한 변경 이벤트를 Portal이 발행하면 Kube가 구독해 로컬 캐시를 즉시 무효화합니다. '권한 변경 이벤트 + TTL 캐시'로 정합성과 가용성을 동시에 얻는 게 정석입니다."

**읽기/쓰기 분리 + 권한 변경 이벤트 구독**을 키워드로 고정하면, 동일 질문이 나와도 "CAP 트레이드오프를 세밀하게 분해한다"로 답할 수 있다.

또한 면접관에게 "다른 방법이 뭐죠?"라고 역질문한 부분도 약점이다. 면접 자리에서 면접관에게 답을 요구하면 그 자체로 준비 부족 신호가 된다. "저는 이렇게 접근할 것 같습니다" + "혹시 더 나은 방향이 있다면 피드백 시간에 듣고 싶습니다"로 끝내야 한다.

## 3. 헥사고날 — 1 인터페이스 1 함수

### Q. Member 서비스의 포트/어댑터 구성?

- **InPort**: 하나의 인터페이스에 **하나의 함수**. 구현체는 UseCase
- **OutPort**: 마찬가지로 하나의 함수. 구현체는 OutAdapter
- **InAdapter**: Controller, Redis Stream Consumer 등 요청이 들어오는 레이어
- **OutAdapter**: 외부 API 호출 (Keycloak 호출, Redis에 토큰 저장 등)
- **원칙 근거**: SRP + 코드 가독성

### Q. 인터페이스 수가 많아지면?

도메인 기준으로 패키지 분리 + 네이밍 규칙(`GetMemberInPort.kt`, `ListMemberInPort.kt` 같은 Verb + Entity 형태)으로 관리. 요즘은 AI에 역할 설명만 주면 파일 경로를 찾아줘서 파일 수가 많은 부담은 크지 않다.

### Q. 한 UseCase가 여러 OutPort를 주입받는 경우 의존성 폭발?

많아도 5개 이하. 크게 문제로 느끼지 않았다.

### 보완 포인트

"1 인터페이스 1 함수" 원칙은 **Interface Segregation Principle (ISP)**의 극단적 적용이다. 답변 라인:

> "1 인터페이스 1 함수는 ISP의 엄격한 적용입니다. 장점은 UseCase가 필요한 기능만 의존한다는 점, 단점은 인터페이스 수 폭발입니다. 저희는 Verb + Entity 네이밍 규칙으로 관리했고, 5개 이하로 주입되는 수준이라 의존성 폭발 문제는 없었습니다. 더 커졌다면 한 도메인의 여러 OutPort를 하나의 'Facade OutPort'로 묶는 것도 가능하지만, 저희 규모에선 불필요했습니다."

`ISP의 엄격한 적용` + `Facade OutPort`라는 키워드를 박아두면 헥사고날 질문이 한 단계 올라간다.

"AI가 찾아준다"는 답변은 **면접에서 쓰면 안 되는 표현**이다. "파일이 많아도 도메인 패키지 분리와 네이밍 규칙으로 탐색 비용이 낮다"로 끝내야 한다.

## 4. Keycloak + OAuth2 Authorization Code Flow + PKCE

### Q. 로그인 흐름을 처음부터 끝까지?

1. 프론트가 백엔드에 로그인 요청
2. 백엔드가 Keycloak redirect URL 반환 (state + PKCE code challenge 포함)
3. 프론트가 redirect URL로 이동 → Keycloak 로그인
4. Keycloak이 `code` + `state`를 프론트로 redirect
5. 프론트가 `code`를 백엔드로 전달
6. 백엔드가 `state` 검증 + Redis에 저장해둔 `code verifier`를 꺼내 Keycloak에 token 교환 요청
7. Keycloak이 access token + refresh token 반환
8. 백엔드가 토큰을 프론트에 전달, 프론트는 스토리지에 저장 후 Authorization 헤더에 넣어 요청

### Q. 왜 백엔드 Redis에 code verifier 저장?

PKCE의 code verifier는 redirect URL 발급 시 생성되고, 이후 code를 token으로 교환할 때 다시 필요해서 세션간 보관이 필요하다.

### 보완 포인트

답변은 정확하다. 한 줄 강화로 PKCE의 존재 이유까지 박아두자:

> "PKCE는 Authorization Code의 가로채기 공격(interception)을 막기 위해 RFC 7636에서 정의된 확장입니다. Client가 `code_verifier`를 만들고 그 해시(`code_challenge`)를 먼저 전송한 뒤, token 교환 시점에 원본 verifier를 다시 제출해 Keycloak이 해시와 매칭해야만 token을 발급합니다. 공격자가 redirect된 code를 가로채도 verifier가 없으면 token을 얻을 수 없습니다. 저희는 BFF 패턴으로 verifier를 백엔드 Redis에 저장해 프론트에 노출되지 않도록 했습니다."

`RFC 7636`, `code_challenge`, `BFF 패턴` 세 키워드를 박아두면 OAuth 질문이 단단해진다.

### Q. 왜 `state`가 필요한가?

CSRF 방지. 악의적 사이트가 사용자를 특정 redirect URL로 유도해 공격자 계정으로 로그인시키는 login CSRF를 막는다.

## 5. 모노레포 공통 모듈의 영향 범위

### Q. 공통 모듈 변경이 다른 서비스를 깨뜨리지 않는다는 걸 어떻게 보장?

- **잘게 쪼개기**: JPA / 인증 / 유틸 등을 별도 모듈로 분리해 변경 영향 범위 최소화
- **설정 위주 코드만**: 공통 모듈이 함수/클래스를 노출하기보다 의존성 추가만으로 설정이 주입되는 방향
- **전체 CI**: 공통 모듈 변경 시 모든 서비스의 테스트가 자동 실행

### Q. 공통 모듈 변경으로 실제 테스트가 깨진 경험?

없었다.

### 보완 포인트

답변은 방어 기법이 3종 제시되어 괜찮다. 한 줄 추가:

> "공통 모듈 변경의 영향 범위 제한은 '설정 주입 중심의 공통 모듈 + 세분화된 모듈 경계 + 전체 CI'로 묶습니다. 여기에 더해 공통 모듈 버전을 의미론적 버저닝으로 관리하고, breaking change는 이름 공간을 바꾼 새 모듈을 추가해 점진적 이행을 허용하는 게 큰 팀에선 필요합니다."

`semantic versioning` + `점진적 이행`을 박아두면 모노레포 거버넌스 질문에 강해진다.

## 6. Kube API vs DB의 경계

### Q. Kube 관련 데이터는 DB에 저장하는가?

Kube API에서 직접 가져오고 DB로 관리하지 않는다. DB에는 **Kube에 없는 우리 서비스 도메인**(NS에 할당된 멤버 같은 것)만 저장.

### Q. NS 목록 조회 시 Kube API + DB를 어떻게 합치나?

OutPort 2개 사용. 하나는 Kube 호출, 하나는 DB 조회, UseCase에서 합쳐서 응답.

### Q. Kube API 응답 지연/타임아웃 대응은?

따로 처리한 적 없다.

### 보완 포인트

Kube API를 "source of truth"로 두는 설계 자체는 정석이다. 다만 다음 한 줄이 빠지면 면접관이 바로 파고든다:

> "Kube API를 source of truth로 두면 데이터 중복 문제가 없지만, 모든 조회가 외부 호출로 나가므로 latency와 장애 격리가 취약합니다. 대응은 (1) Kube API 호출에 타임아웃 + 서킷 브레이커, (2) 조회 결과를 short-TTL 캐시(예: 30초)로 감싸 burst 트래픽 흡수, (3) list 응답은 watch 이벤트 기반으로 로컬 상태를 유지하는 Informer 패턴이 정석입니다."

`Informer 패턴`을 한 번 박아두면 Kube API 답변이 즉시 한 단계 올라간다.

## 전체 평가

### 강점

- **BC 분리 기준** — "쿠버네티스를 다루는가 vs 우리 도메인을 다루는가"로 명확히 설명
- **권한 캐싱 배제 이유** — 정합성 리스크를 삭제 권한 예시로 구체적으로 설명
- **PKCE + state 흐름** — OAuth2 Authorization Code Flow의 단계별 흐름을 정확히 풀어냄
- **헥사고날 포트/어댑터 In/Out 구분** — 1 인터페이스 1 함수 원칙과 근거를 명확히 답변
- **공통 모듈 설정 주입 방식** — 함수 노출 대신 의존성 추가만으로 설정이 들어오도록 설계

### 약점 패턴

- **면접관에게 역질문** — "혹시 다른 방법이 있을까요?" 답변은 면접에서 위험. 준비 부족 신호
- **읽기/쓰기 권한 분리 미검토** — 면접관이 직접 짚은 부분을 "검토 안 했다"로 끝냄
- **AI가 찾아준다** — 파일 탐색 답변에 AI를 언급. 면접에서 쓰면 안 되는 표현
- **"프라이빗 클라우드라 사용자 수가 적어서 문제 없었다"** — 트래픽 전제를 방패로 쓰는 답변이 반복됨. "문제 없었지만, 트래픽이 커지면 X" 형태로 항상 이어 붙여야 함
- **Kube API 장애 대응 없음** — source of truth가 외부에 있는데 장애 격리를 고려 안 한 건 MSA 설계에서 중요한 누락

### 다음 모의 면접 전 정리할 것

1. **권한 분리 설계 라인** — 읽기(TTL 캐시 + stale 허용) vs 쓰기(동기) + 권한 변경 이벤트 구독
2. **OAuth2 키워드 3종** — `RFC 7636`, `code_challenge`, `BFF 패턴`
3. **Kube API 답변 라인** — Informer 패턴 + short-TTL 캐시 + 서킷 브레이커
4. **면접 역질문 금지 룰** — 면접에서 "뭐가 답인가요"는 절대 금지. "제 접근은 X, 피드백은 이후에"로 끝내기
5. **"트래픽이 적어서"로 끝나는 답변 금지** — 항상 "적어서 OK였지만, 커지면 Y"로 이어 붙이기
6. **"AI가 찾아준다" 금지** — 파일 탐색은 도메인 패키지 + 네이밍 규칙으로만 설명
7. **헥사고날 ISP 프레이밍** — 1 인터페이스 1 함수는 ISP 엄격 적용 + Facade OutPort 대안
