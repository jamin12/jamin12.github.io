이력서의 "대용량 데이터 다운로드" 항목으로 한 세션을 끝까지 추궁당한 모의 면접 기록. JPA 1차 캐시, `flush` vs `clear` 정정, 트랜잭션 경계, 스레드풀 거부 정책, 커버링 인덱스의 정의까지 한 주제를 깊게 들어갔다. 답변 도중 면접관과 두 번 충돌해서 "범위 밖이다", "왜 이해 못 하냐"로 끊은 부분도 그대로 기록한다.

---

## 0. 이력서 요지

- **문제**: 2GB 메모리 서버에서 1000만 건 CSV 다운로드 → OOM
- **사전 검토**: 사용자 선택에 따라 컬럼/날짜/관제점이 바뀌어 미리 파일을 만들어둘 수 없음
- **단계별 해결**:
  1. JPA Stream 조회로 전환 → OOM 미해결
  2. 날짜 단위 호출 쪼개기 + 쿼리 분리 + 커버링 인덱스
  3. `OutputStreamWriter` + `StreamingResponseBody` 응답 스트리밍
  4. 영속성 컨텍스트 `clear`로 캐시 제거
  5. 동시 요청 수 제한용 `ThreadPoolTaskExecutor`
- **결과**: 대용량에서 안전하게 운영되는 다운로드 서버 구축

## 1. 1단계 Stream 전환이 OOM을 못 푼 이유

### Q. Stream으로 바꿨는데도 OOM이 난 정확한 원인?

두 가지가 겹쳤다고 본다.

1. **MySQL JDBC의 Stream 동작 특성**: 기본적으로 MySQL JDBC는 `setFetchSize(Integer.MIN_VALUE)` 같은 옵션을 명시하지 않으면 결과를 클라이언트 메모리에 전부 올린 뒤 stream을 흘려준다. row 단위로 진짜 흘리려면 추가 설정이 필요하고, 당시에는 거기까지 검증하지 않았다. 또 stream을 흘리는 동안 커넥션을 계속 잡아두는 부작용도 있어서 날짜 단위로 쪼개는 방향이 결과적으로 더 나았다.
2. **JPA 1차 캐시(영속성 컨텍스트)**: 조회된 엔티티가 트랜잭션 동안 컨텍스트에 누적되며 메모리를 잡아먹었다.

### 보완 포인트

답변은 정확하지만 "테스트는 안 해봤다"가 즉답에 들어간 게 약점. 다음 면접 답변 라인:

> "MySQL JDBC는 기본적으로 결과셋을 클라이언트 버퍼에 전부 올린 뒤 흘립니다. row 단위 스트리밍을 강제하려면 `useCursorFetch=true` + `setFetchSize`를 적절히 설정해야 합니다. 다만 cursor fetch는 커넥션을 길게 잡고, 트랜잭션도 길어지므로 결국 우리 케이스에서는 날짜 분할 + 짧은 트랜잭션이 더 안전한 선택이었습니다."

`useCursorFetch`를 키워드로 박아두면 즉답에서 흔들리지 않는다.

## 2. 영속성 컨텍스트와 OOM

### Q. 여기서 말하는 "영속성 캐시"가 정확히 뭐?

JPA의 1차 캐시 = 영속성 컨텍스트. DB에 들어가거나 조회되기 전후로 엔티티를 관리하는 영역. "영속화되기 전 단계의 엔티티가 살아 있는 곳"이라고 보면 된다.

### Q. 1000만 건 조회에서 영속성 컨텍스트가 어떻게 OOM을 만들었나?

엔티티가 조회될 때마다 1차 캐시에 저장된다. 캐시는 메모리이고, 1000만 건의 엔티티 객체를 들고 있으면 heap이 터진다.

### Q. 어떻게 제거했나?

날짜 범위로 잘라서 조회한 뒤 `EntityManager.flush()`를 호출해 ...

→ 면접관: "조회 흐름에서 flush가 적절한가? flush의 역할이 뭔지?"

**정정**: `flush`는 변경감지(dirty checking) 후 변경된 엔티티의 DML을 DB로 내보내는 것. 내가 실제로 호출한 건 **`clear()`**. clear는 영속성 컨텍스트의 모든 엔티티를 detach시켜 메모리에서 떼어낸다.

### Q. clear 호출 후 detach된 엔티티를 OutputStream에 쓰는 흐름에서 문제는 없었나?

`StreamingResponseBody`로 비동기 OutputStream에 직접 쓰고, 주기적으로 `flush()`로 클라이언트에 내보냈다. detach 이후에는 그 엔티티의 lazy 필드를 더 따라가지 않으므로 미리 필요한 필드만 읽고 clear한 뒤 쓰는 흐름이라 문제는 없었다.

### 보완 포인트

`flush` ↔ `clear`를 입에서 한 번 헷갈린 게 가장 약했다. 둘의 책임을 다음 한 줄로 외워두자:

| 메서드 | 책임 | 영속성 컨텍스트 비우는가 |
|--------|------|--------------------------|
| `flush()` | 변경감지 + DML을 DB로 내보냄 | 아니오 |
| `clear()` | 모든 엔티티를 detach | 예 |
| `close()` | 컨텍스트 자체를 닫음 | 예 (영구) |

답변 라인:

> "1000만 건 조회 흐름에서는 `clear()`로 1차 캐시를 비웠습니다. `flush`는 DML 동기화이고 조회 흐름에는 의미가 없습니다. 또한 처음부터 `Hibernate.StatelessSession` 같은 상태 비저장 세션을 쓰는 것도 대안이지만, 우리는 일반 세션을 유지한 채 청크별 `clear` 전략으로 갔습니다."

`StatelessSession`을 키워드로 깔면 면접관이 "다른 대안 알아?"로 들어와도 한 발 앞서 있다.

## 3. 트랜잭션 경계와 데이터 일관성

### Q. 날짜 분할 → clear → OutputStream 흐름이 한 트랜잭션이었나?

read만 하는 작업이라 stream 작업을 열 때마다 트랜잭션을 따로 잡지 않았다. 다만 지금 다시 한다면 청크별로 짧은 트랜잭션을 명시적으로 두는 게 더 안전했을 것 같다.

### Q. 트랜잭션 없이 조회하는 도중 다른 사용자가 데이터를 수정하면?

이 데이터는 insert만 되고 update/delete가 절대 없는 특성이라 그 시나리오 자체가 발생하지 않는다. 만약 수정이 있다면 최신 버전이 그대로 읽힐 것이다.

### 보완 포인트

데이터 특성으로 정합성 문제를 막은 건 좋은 답변이다. 더 강한 답변 라인:

> "이 테이블은 append-only이므로 phantom read가 비즈니스적으로 무해합니다. 만약 update가 가능한 테이블이었다면 청크별로 `REPEATABLE READ`나 스냅샷을 잡아야 하지만, append-only에서는 그 비용을 낼 필요가 없습니다."

"왜 안 했나"가 아니라 "데이터 모델이 그 비용을 면제해줬다"로 정리하면 시니어 답변.

## 4. 스레드풀과 거부 정책

### Q. 스레드풀로 어떻게 제한했나?

`ThreadPoolTaskExecutor` 사용. 한도 초과 시 큐에 대기하도록 했다.

### Q. 큐 사이즈 제한은? 큐도 가득 차면?

큐를 일부러 안 썼다. 스레드는 2개로 고정. 메모리를 많이 쓰는 작업이라 의도적으로 큐 없이 갔다. `CallerRunsPolicy`도 메모리 부담 때문에 의도적으로 안 썼다.

### Q. 큐도 없고 스레드 2개 고정인데 3번째 요청이 오면?

당시에는 사용 빈도가 낮아 별도 처리를 안 했다. 비동기 응답이라 일반 ControllerAdvice/에러 핸들러로는 잡히지 않을 것이다. 다음에 한다면 비동기를 시작하기 전에 풀의 가용 슬롯을 확인하고 "대기해주세요" 같은 사용자 메시지를 먼저 내려주는 게 맞다.

### 보완 포인트

`ThreadPoolExecutor`의 거부 정책 4종을 한 번 정리:

| 정책 | 동작 |
|------|------|
| `AbortPolicy` (기본) | `RejectedExecutionException` 던짐 |
| `CallerRunsPolicy` | 호출한 스레드가 직접 실행 (역압) |
| `DiscardPolicy` | 조용히 버림 |
| `DiscardOldestPolicy` | 큐에서 가장 오래된 작업 버리고 새 작업 추가 |

답변 라인:

> "스레드 2개 + 큐 0, 거부 정책은 `AbortPolicy`로 두고, 비동기 시작 전에 `getActiveCount() < maxPoolSize`를 체크해 거절 케이스를 사용자 친화적인 HTTP 응답(예: 429 Too Many Requests + Retry-After)으로 미리 매핑합니다. CallerRunsPolicy를 안 쓴 이유는 호출 스레드가 톰캣 워커이고, 거기서 메모리 무거운 작업을 직접 돌리면 톰캣 풀까지 잠겨서 다른 API가 같이 죽기 때문입니다."

"왜 CallerRunsPolicy를 안 썼는지"를 명시적으로 설명할 수 있어야 한다.

## 5. 커버링 인덱스 — 잘못 적은 용어

### Q. 커버링 인덱스를 적용했다고 했는데, 인덱스 구성과 그 이유는?

이력서에 "커버링 인덱스"라고 적었지만 공부하면서 정확한 정의(SELECT 컬럼까지 인덱스에 포함)를 알게 됐다. 실제 인덱스는 `(관제점, 날짜)` 복합 인덱스이고, SELECT에 `관제값`이 포함되므로 **엄밀히는 커버링 인덱스가 아니다**.

### Q. 그러면 인덱스 스캔 후 관제값을 가져오기 위해 어떤 추가 작업?

인덱스 스캔 후 해당 row의 관제값을 가져오기 위해 **테이블 랜덤 액세스**가 발생한다.

### Q. 1000만 건 규모에서 랜덤 액세스 비용을 알면서도 진짜 커버링 인덱스로 안 만든 이유?

`(관제점, 날짜, 관제값)` 형태로 관제값을 인덱스 끝에 추가하면 커버링이 된다. 다만 트레이드오프가 있다:

- 1000만 건 + 계속 들어오는 INSERT 워크로드
- 인덱스에 컬럼이 추가되면 인덱스 크기 증가 → INSERT 성능 저하 + 메모리에 인덱스를 유지하는 비용 증가
- 조회 빈도 vs 쓰기 빈도를 비교했을 때 쓰기가 압도적으로 많음

→ 최종적으로는 **인덱스 추가 대신 집계 테이블 도입**으로 풀었다.

### 보완 포인트

이 섹션은 "이력서에 잘못된 용어를 썼다"는 사실을 면접에서 인정해야 했다. 다음에 이력서를 다듬을 때:

- "커버링 인덱스" → "복합 인덱스 (관제점, 날짜)"로 정정
- "관제점/날짜 복합 인덱스로 필터 단계의 row 수를 줄이고, SELECT의 관제값은 테이블 룩업으로 처리. 진짜 커버링 인덱스는 INSERT 부담 때문에 채택하지 않고 집계 테이블로 우회"

이력서 단어 하나가 면접에서 5분을 잡아먹는 사례. 이력서는 코드가 아니어도 정확해야 한다.

## 6. 면접관과의 마찰 (자기반성)

이번 세션에서 두 번 면접관에게 짜증을 냈다.

1. **사용자 다운로드 흐름 설계 질문에서**: "이거 너무 범위 밖이야" — 이력서 항목에서 람다/S3 대안으로 답이 흐른 뒤, 면접관이 그 흐름을 자세히 물었을 때 "범위 밖"으로 끊었음
2. **커버링 인덱스 질문에서**: "아까 말했잖아 왜 이해 못 하냐" — 면접관이 인덱스 끝에 컬럼만 추가하는 방식을 제안했는데, 내가 그걸 "관제값 기준 인덱스"로 오해해서 같은 답을 반복

### 진짜 원인

두 경우 모두 **내가 답변을 충분히 정확히 안 깐 게 원인**이다.

- 사례 1: "람다 + S3" 대안을 즉석에서 던졌으면 그 흐름의 책임을 끝까지 져야 한다. 던져놓고 "범위 밖"으로 도망치면 전체 답변 신뢰가 무너진다.
- 사례 2: "관제값을 인덱스에 추가하는 두 가지 방식(조건절용 vs SELECT 커버링용)"을 분리해서 설명했어야 한다. 분리 안 하니 면접관 질문이 같은 단어로 들렸고, 내가 "왜 같은 걸 또 묻냐"로 받아들였다.

### 다음 면접 전 약속

1. **즉석 대안을 던졌으면 그 흐름의 끝까지 책임진다.** 모르면 "거기까지는 설계해보지 않았다"로 정직하게 끊고, "범위 밖"이라는 단어는 쓰지 않는다.
2. **면접관이 같은 질문을 반복하면 내가 잘못 답한 것**이라고 가정하고, 답변을 다시 풀어 설명한다. 짜증은 절대 금지.
3. **답변에 한정 절(`단, X 조건에서는 Y`)을 항상 붙인다.** "관제값 인덱스"가 두 가지 의미가 될 수 있으면, 처음부터 두 의미를 분리해서 답한다.

## 7. 메모리 병목 진단 도구

### Q. 메모리 문제를 로그 대신 어떻게 찾을 건가?

지금이라면 메트릭(JVM heap, GC, DB 커넥션 p95/p99)을 본다. Prometheus + Grafana로 수집/시각화한 경험이 있다.

### Q. 이 상황에서 heap 메트릭에서 어떤 패턴을 봤나?

요청이 들어올 때 heap 사용량이 갑자기 치솟는 패턴.

### Q. 그게 1차 캐시 때문인지 다른 원인인지 어떻게 구분?

→ 면접관이 같은 질문을 반복해서 짜증으로 받음. 사실 이건 답변 못 한 부분.

### 보완 포인트

heap 치솟음이 어디서 왔는지 정확히 분리하려면 다음 도구가 필요하다:

- **Heap dump + Eclipse MAT / VisualVM**: dominator tree로 어떤 객체 그래프가 메모리를 잡고 있는지 확인. JPA 1차 캐시면 `PersistenceContext` 또는 `EntityEntry`가 dominator로 잡힌다
- **JFR (Java Flight Recorder)**: 시간대별 객체 할당 + GC 이벤트 추적
- **GC 로그 + GCViewer**: Young/Old 비율과 promotion 패턴

답변 라인:

> "메트릭으로는 heap이 치솟는 시점만 알 수 있고, 원인 객체를 특정하려면 Heap dump를 떠서 MAT의 dominator tree로 확인해야 합니다. 1차 캐시가 원인이면 보통 `org.hibernate.engine.spi.PersistenceContext` 하위에 `EntityEntry`들이 메모리의 큰 비중을 차지하는 형태로 잡힙니다."

이 한 문단이면 "기반 지식으로 추론했다"가 "도구로 검증할 수 있다"로 한 단계 올라간다.

## 전체 평가

### 강점

- **MySQL JDBC stream 동작 특성 인지** — 1단계가 왜 안 풀렸는지 정확한 원인 두 가지 짚음
- **영속성 컨텍스트 정의** — 1차 캐시 = 영속성 컨텍스트, 메모리 누수 메커니즘을 흐름으로 설명
- **트레이드오프 인식** — 인덱스 추가 vs 집계 테이블에서 INSERT 빈도 우위를 정확히 판단
- **append-only 데이터 특성 활용** — 트랜잭션 없이 조회해도 정합성 문제가 없는 이유를 데이터 모델로 설명
- **자기 약점 인정** — 이력서의 "커버링 인덱스" 표현이 잘못이라는 걸 면접 도중 솔직히 인정

### 약점 패턴

- **`flush` ↔ `clear` 헷갈림** — JPA 핵심 API에서 한 번 헷갈린 건 가장 치명적
- **이력서의 잘못된 용어** — "커버링 인덱스"가 면접의 5분을 잡아먹음. 이력서는 정확해야 함
- **즉석 대안의 끝까지 책임지지 못함** — 람다/S3를 던져놓고 "범위 밖"으로 끊은 건 신뢰를 무너뜨림
- **면접관과 마찰 두 번** — 짜증은 어떤 경우에도 금지. 같은 질문이 두 번 오면 내 답변을 의심
- **"테스트 안 해봤다", "메모리 안 측정했다"** — 검증 없는 답변이 여전히 자주 등장
