이력서의 "이벤트 기반 분산 트랜잭션 구축" 항목으로만 한 세션을 끝까지 추궁당한 모의 면접 기록. Redis Stream 내부 동작에서 시작해 Outbox/Saga, 멱등성, 동시성, 비동기 위임의 ACK 타이밍까지 한 주제만 파고들었다. 같은 답을 두 번 놓쳐서 면접관(AI)을 두 번 멈춰 세웠고, 그게 면접관 탓이 아니라 내 답변 베이스라인 설계의 문제였다는 자기 회고도 같이 적는다.

## 0. 이력서 요지 (한 줄 요약)

- **문제**: 유스케이스 내부 롤백 → 복잡도 폭발, Template 공통코드도 롤백 실패 시 추적 불가
- **전환**: 동기 보상 → 이벤트 기반 비동기
- **인프라**: Kafka/RabbitMQ 대신 기존 Redis 활용, Pub/Sub의 유실·중복 → Redis Stream + Consumer Group
- **원자성**: Outbox 패턴 (비즈니스 로직 + 이벤트 저장 = 한 트랜잭션)
- **테이블 분리**: Status/Event 분리 — 이력 보존과 폴링 성능 충돌 회피
- **Saga**: 중앙 오케스트레이션 → 코레오그래피 전환, 전용 파일 70개 제거, 구독자 17 → 3개

## 1. Redis Stream 내부 동작

### Q. Redis Stream이 정확히 어떤 자료구조이고 내부적으로 어떻게 동작?

시간순으로 INSERT만 되는 자료구조. 여러 Consumer Group을 가질 수 있고, 그룹별 last delivered ID를 기준으로 어디까지 읽었는지 추적해 그룹 내 컨슈머끼리 메시지를 분배한다.

### Q. 메시지가 "처리 완료"됐다는 걸 Stream은 어떻게 인식?

소비자가 처리 완료 후 ACK(`XACK`)를 보내면 처리 완료로 인식.

### Q. ACK 보내기 전에 컨슈머가 죽으면?

미처리 상태로 남는다. 다만 비즈니스적으로 문제가 없는 이유는, 다음 스텝이 진행되려면 결과 이벤트가 발행되어야 하는데, 결과 이벤트 발행 전에 컨슈머가 죽으면 발행이 안 되니 시작한 쪽(state)에서 미완료를 감지하고 재발행으로 복구한다.

---

## 2. 재발행 트리거와 코레오그래피의 모순

### Q. ACK 없이 죽은 메시지의 재처리는 누가, 언제 트리거?

비즈니스 로직이 끝난 후 "완료 이벤트"가 시작한 곳(진입점)으로 발행된다. 이 이벤트가 발행되지 않으면 시작한 곳에서 완료 처리가 안 되고, 미완료 상태를 보고 재발행한다. 이건 ACK와는 별개 레이어다.

### Q. 코레오그래피라면 "시작한 곳"이 전체를 모니터링하고 재발행을 트리거하는 건데, 이건 사실상 오케스트레이터 아닌가?

맞다. 사실 이게 조금 섞여 있다. **코드는 코레오그래피, DB는 오케스트레이션**으로 구현돼 있다.

- 코드 레벨: 각 UseCase는 자기가 성공했다는 사실만 이벤트로 날리고 끝난다. 어떤 스텝이 어디로 가야 하는지 본인은 모름.
- DB 레벨: 시작한 곳의 state 1행이 saga 끝날 때까지 모든 진행 상태를 들고 있음.

### 보완 포인트

면접관이 정확히 짚었다. **"코레오그래피로 전환했다"고 단언하면 즉시 모순이 잡힌다.** 하이브리드라고 처음부터 정리해야 한다:

- **이벤트 흐름**: 코레오그래피 (각 UseCase는 다음 스텝을 모름)
- **상태 추적**: 중앙 집중 (state 테이블 1행이 saga 전체 상태)

이렇게 답하면 "결합도 낮음(코레오그래피) + 가시성(오케스트레이션)"을 동시에 가진 의도된 설계로 들린다.

## 3. 4개 테이블 구조

처음 이력서에는 2개로 적었지만 시행착오 후 4개로 늘었다.

| 테이블 | 역할 | 핵심 컬럼 |
|--------|------|-----------|
| **outbox** | 발행 관리 | event_id, event_name, step, payload, domain, domain_id, published, retry_count |
| **state** | 진행 상태 추적 | event_id, event_name, current_step, accumulated_payload, retry_count, status |
| **log** | 이력 기록 | event_id, event_name, step, status, error |
| **idem** | 멱등성 체크 | event_id, step (복합 unique) |

### Q. outbox와 state는 둘 다 event_id, name, step, payload를 가지는데 역할 차이는?

- **outbox**: 발행 단위. 한 이벤트의 각 스텝마다 INSERT. payload는 그 스텝만의 데이터.
- **state**: 상태 단위. 한 이벤트당 행 1개를 UPDATE. payload는 스텝별로 누적. 롤백 시 누적 payload를 사용해야 하기 때문.

### Q. 3번째 스텝에서 실패하면 보상은 어떻게?

try-catch에서 catch에 진입하면 시작한 이벤트 쪽으로 실패 이벤트를 보낸다. 시작한 곳은 실패 응답이 오면 state에서 누적된 payload를 가져와 역순으로 보상 이벤트를 발행한다.

---

## 4. 실패 이벤트 자체가 유실되면

### Q. 실패 이벤트가 시작한 곳에 도달하기 전에 컨슈머가 죽으면?

결과가 안 오므로 재시도 횟수까지 재발행. 그래도 안 되면 알람 + 수동 처리. 다만 **알람은 아직 미구현**.

### Q. 재시도 초과 후의 수동 트리거는 누가?

자동화 없음. 사람이 알림을 보고 재시도 API를 호출하는 식. 현재 미구현, 테스트 부족.

---

## 5. 스케줄러 스케일 아웃 & 멱등성 동시성

### Q. Outbox 폴링 스케줄러를 여러 인스턴스로 스케일 아웃하면?

ShedLock이나 Quartz로 분산 락을 걸 수 있는 걸 알지만 적용하지 않았다. **중복 발행을 전제로 두고 컨슈머 측 멱등 처리**로 풀고 있다. 다른 비즈니스 우선순위가 높아 미처리.

### Q. 두 컨슈머 인스턴스가 동시에 idem INSERT를 시도하는 찰나의 경합은?

DB의 unique 제약조건이 동시성을 보장한다. DBMS가 동시 INSERT를 직렬화하므로 둘 중 하나는 duplicate 에러가 난다.

### Q. duplicate 에러가 난 쪽은?

에러를 무시하고 비즈니스 로직 실행 없이 즉시 종료.

### Q. (놓친 부분) duplicate로 막혔는데 먼저 INSERT한 쪽이 비즈니스 로직 도중에 죽으면 idem만 남고 재시도가 막히지 않나?

→ **이미 답변한 내용**: idem INSERT + 비즈니스 로직 + outbox 이벤트 저장이 **하나의 트랜잭션**. 중간에 죽으면 idem까지 같이 롤백된다. 면접관이 한 번 놓쳤고 짚어주자 인정했다.

---

## 6. 외부 API의 멱등성

### Q. idem + 비즈니스 로직 + outbox를 한 트랜잭션으로 묶었는데, 비즈니스 로직 안에 외부 API 호출이 있다면?

DB라면 트랜잭션에 묶여서 같이 롤백되지만 외부 API는 롤백 불가. 대응:

1. API 자체가 멱등성 키를 받도록 제공
2. 여러 번 호출해도 무해한 경우
3. 중복 저장 시 unique 위반으로 두 번째가 거절되도록 (예: A 저장 후 다시 A 저장 시 duplicate 에러)

### Q. 외부 API가 멱등성 키도 없고 중복 저장도 막지 않으면?

따로 처리하지 않고 있다. 즉석 답변:

- 호출 직전에 "이 작업을 시도 중"이라는 마킹을 DB에 PENDING으로 저장
- 호출 성공 후 완료 처리
- 완료 처리 전에 죽으면 PENDING이 남음
- 비즈니스 로직 특성에 따라 분기:
  - **재실행 안전** → 다시 시도
  - **재실행 위험** → PENDING 상태로 두고 재시도 차단, 로그 남기고 수동 확인

### 보완 포인트

방향은 맞다. 정식 용어로는 **2단계 마킹(intent log)** 패턴이고, 분산 락 + 사람 개입 큐의 조합으로 정리하면 더 매끄럽다. 답변 라인:

> "협조하지 않는 외부 API는 본질적으로 At-most-once 보장이 안 되므로 우리 쪽에서 멱등성을 만들 수 없습니다. 그래서 작업 의도(intent)를 먼저 영속화하고, 사람이 들어가는 출구를 미리 열어둡니다."

## 7. 구독자 3개 구조와 향후 개선

### Q. 17개 → 3개로 줄었는데, 3개의 구독자는 각각 무엇?

이전에는 스텝별로 구독자가 있어서 각각 Consumer Group을 가졌다. 지금은 **메서드별이 아닌 이벤트별 그룹**으로 묶었다.

### Q. 3개의 기준은?

Saga 흐름 단위. 예를 들어 K8s NS 생성 + 회원 생성 같은 흐름 하나당 구독자 1개. **현재 Saga 흐름이 3개라 구독자도 3개**.

### Q. Saga 흐름이 늘어나면 구독자도 그만큼 늘어나는 구조?

맞다. 다만 공부하면서 알게 된 더 나은 방향: **서버당 Consumer Group 1개**로 통합하고, 받은 이벤트를 코드 레벨 분기로 라우팅하는 게 낫다.

### Q. 그렇게 하면 한 흐름의 처리 지연이 다른 흐름까지 블로킹할 텐데?

컨슈머가 메시지를 받자마자 **비동기 위임**한다. Redis Stream의 추가/읽기는 O(1)이므로 stream 자체는 병목이 아니고, 처리 속도는 컨슈머 측 비동기 처리로 풀린다.

## 8. 비동기 위임의 ACK 타이밍 (면접관이 같은 질문을 두 번 한 부분)

### Q. 비동기 위임 시 ACK는 언제?

위임 직후 즉시 ACK. 재처리 로직은 이벤트 DB(outbox/state)가 책임지므로 Stream 차원의 재전달은 필요 없다.

### Q. 비동기 위임받은 스레드가 idem INSERT 전에 죽으면? ACK는 이미 나갔고 idem에도 기록이 없는데?

→ **이미 답변한 내용**: idem INSERT + 비즈니스 로직 + outbox 이벤트 저장이 한 트랜잭션이라, 트랜잭션 진입 전에 죽으면 outbox에 결과 이벤트가 남지 않는다. 시작한 곳의 state는 미완료로 남고, **outbox 폴링 스케줄러가 재발행**한다. 즉 Stream 재전달이 아니라 **outbox 재발행이 안전망**.

면접관이 두 번째로 같은 내용을 다시 물어와서 "집중 좀..."으로 끊었다.

### 보완 포인트 (자기반성)

면접관이 놓친 게 이상한 게 아니다. **답변할 때 핵심 안전망(outbox 재발행 = 진짜 source of truth)을 충분히 강조하지 않은 것**이 진짜 원인이다. 다음에는 첫 ACK 질문에서부터 다음 한 문단을 깔아두자:

> "ACK는 위임 시점에 보냅니다. Stream의 ACK는 의미상 '메시지를 받았다'로만 사용하고, **실제 진행 보장은 outbox 테이블이 source of truth**입니다. 비동기 작업이 어디서 죽든, state가 미완료면 outbox 폴링이 새 이벤트를 다시 만들어 재발행합니다. Stream 재전달과 outbox 재발행은 별개 레이어이고 우리는 후자에 의존합니다."

이 한 문단이 깔리면 이후 모든 "죽으면 어떻게 됨?" 질문이 같은 답으로 수렴해서 면접관도 추적이 쉬워지고, 같은 답을 반복할 일도 없다.

## 전체 평가

### 강점

- **4개 테이블의 책임 분리**를 정확히 설명, 특히 outbox/state의 차이를 "스텝마다 INSERT vs 행 1개를 UPDATE + payload 누적"으로 명확히 구분
- **하이브리드 구조 인정** — 코레오그래피/오케스트레이션이 섞여 있다는 사실을 면접관 지적 후 즉시 인정하고 코드/DB 레이어로 분리해 설명
- **idem + 비즈니스 + outbox = 1 트랜잭션** 경계 설계가 정확
- **동시성 처리** — DB unique 제약을 활용한 자연스러운 직렬화를 명확히 인지
- **즉석 대안 제시** — 외부 API 비협조 케이스에서 2단계 마킹 + 수동 출구를 즉석에서 설계해 답변

### 약점 패턴

- **"이미 말했잖아요" 두 번** — 면접관이 놓친 부분이지만, 진짜 원인은 처음에 **"outbox가 source of truth"라는 안전망**을 일찍 강조하지 않은 것. 답변 베이스라인을 깔지 않으면 같은 질문이 다른 각도로 계속 들어온다
- **"미구현/우선순위가 낮아 안 했다" 3번** (ShedLock, 알람, 외부 API 멱등성) — "안 했다"에서 끝내지 말고 항상 **"안 했지만 한다면 X"**까지 한 호흡으로 이어 붙여야 한다
- **용어의 정확도** — "코레오그래피로 전환했다"는 단언, "비동기로 보내면 됩니다" 같은 두루뭉술한 답변은 면접관이 즉시 파고든다. **"단, X 조건에서는 Y"** 같은 한정 절을 함께 넣는 연습 필요
- **감정 반응** — "집중 좀..." 두 번. 내용은 맞지만 톤은 "아까 말씀드렸듯이 트랜잭션 경계가 묶여 있어서…"로 다시 풀어주는 게 면접 자리에 안전

### 다음 모의 면접 전 정리할 것

1. **ACK + outbox 폴링의 관계도** 1장 그려두기 (Stream 레이어 / DB 레이어 분리)
2. **하이브리드 Saga 구조 한 문단 설명** 고정
3. **"미구현이지만 한다면 X"** 답변 템플릿 — ShedLock, 알람 큐, 외부 API 멱등성 등 미구현 항목별로 1줄씩 미리 준비
4. **답변 첫 문장에 "source of truth"를 박는 습관** — Stream/DB/외부 시스템이 섞인 질문에서 항상 "진짜 권위는 어디인가?"를 먼저 선언하면 이후 모든 죽음 시나리오가 그 한 점으로 수렴한다
