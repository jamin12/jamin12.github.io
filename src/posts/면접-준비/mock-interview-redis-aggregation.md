이력서의 "집계 효율화" 항목으로 한 세션을 추궁당한 모의 면접 기록. Redis 장애 시 failover, RENAME 스냅샷의 원자성, 배치 ID로 중복/누락 처리, SCAN vs KEYS, Summary 테이블 스키마와 복합 인덱스 순서까지 한 주제를 깊게 들어갔다. "Redis 명령어는 잘 모른다"로 한 번 무너진 뒤 SCAN/KEYS까지 회복한 흐름이 이번 세션의 전환점이었다.

---

## 0. 이력서 요지

- **문제**: 좋아요/싫어요/팔로우/조회수 집계 → 이벤트마다 insert + GroupBy 쿼리 조회. 통계 페이지 10초 초과
- **개선 플로우**:
  - 사용자 이벤트 → Redis에 기록
  - Spring Schedule 1분마다 Redis → DB 이관, Redis 비움
  - Total 컬럼 + Summary 테이블에 스냅샷 저장
- **Redis 선택 이유**: 자주 발생하는 이벤트로부터 DB 리소스 보호, `increment()`로 동시성 문제 해결
- **결과**: 10초 → 20ms

## 1. Redis 기본 + 장애 대응

### Q. Redis란?

In-memory key-value 데이터베이스. 데이터 쓰기/조회가 매우 빠름.

### Q. Redis 프로세스가 죽으면?

인메모리라 안의 내용이 전부 사라진다. 이를 막기 위해:
- 디스크 저장 옵션(RDB/AOF)
- Sentinel 모드, Cluster 모드
- 애플리케이션 쪽 failover 설정

### Q. 이 프로젝트에서 Redis 죽었을 때 어떻게 처리했나?

구현은 안 했다. 만약 적용한다면 사용 용도별로:
- **캐싱 용도**: Java in-memory 캐시로 대체 또는 DB 바로 접근
- **분산 락**: 락 획득 실패로 에러 반환
- **Stream**: 비동기 로직이라 Redis 재생성까지 기다리는 로직

### Q. Spring Schedule이 1분마다 옮기는 구조인데, Redis 죽으면 1분치 데이터가 사라지는 것에 대해?

**비즈니스 영향이 작은 데이터**라 failover를 따로 설정하지 않았다. 우리 서비스는 정치인 정보 제공 서비스고, 조회수는 핵심 비즈니스 로직과 거리가 멀다. 만약 보호가 필요하다면 Sentinel/Cluster로 Redis 자체를 안 죽게 만드는 방향.

### 보완 포인트

답변은 용도별 분리가 명확해서 좋았다. 추가로:

> "Redis 장애 대응은 항상 '어떤 용도로 쓰는가'에서 출발합니다. 캐시면 cache-aside 패턴에서 Null 반환 후 DB fallback, 분산 락이면 fail-fast, 이벤트 큐면 재처리 큐로 우회. 이 프로젝트는 집계 버퍼 용도라 1분치 유실이 허용되고, 허용되지 않았다면 Redis 앞에 Kafka 같은 영속 큐를 한 단계 더 두는 게 맞습니다."

`cache-aside`, `fail-fast` 키워드를 박아두면 용도별 분리가 시니어 답변으로 들린다.

## 2. RENAME 스냅샷과 원자성

### Q. 이관 도중 새 이벤트가 들어오면?

`RENAME`으로 원본 key를 다른 key로 이동(스냅샷)하고, 이후 들어오는 이벤트는 비어 있는 원본 key에 새로 쌓인다.

### Q. RENAME 흐름을 처음부터 끝까지?

1. `RENAME count:{id} snapshot:{id}:{batchId}` — 원본을 스냅샷 키로 이동 (원자적)
2. 스냅샷 키에서 값을 읽어 DB에 저장
3. 스냅샷 키 삭제

### Q. RENAME이 원자적인 근거?

Redis는 싱글 스레드 + 이벤트 루프 기반이라 명령 하나는 큐에 쌓이고 순서대로 실행된다. 하나가 끝나기 전에는 다음이 실행되지 않는다.

### Q. 전체 과정(RENAME → DB 저장 → 스냅샷 삭제)이 원자적인가?

원자적이지 않다. 세 단계 사이에서 장애가 나면 상태가 어긋난다.

### 해결 방안 — 배치 ID

스냅샷 키에 배치 ID를 붙이면(`snapshot:{id}:{batchId}`):
- DB 저장이 실패하고 다음 스케줄이 돌면, **기존 스냅샷 키와 충돌 없이** 새 배치 ID로 스냅샷이 생성됨
- DB에도 배치 ID를 함께 저장해, **중복 insert를 배치 ID로 skip** 가능

### 보완 포인트

이 섹션의 답변은 즉석에서 설계를 만들어가며 풀었다. 본질적으로는 **outbox 패턴의 Redis판**이다. 답변 라인:

> "이 구조는 DB outbox 패턴을 Redis에 적용한 형태입니다. 배치 ID는 멱등성 키이고, DB의 unique 제약으로 중복을 막습니다. DB 저장이 성공하면 Redis 스냅샷을 삭제하고, 실패하면 스냅샷이 남아 다음 스케줄에서 합산됩니다. 서버 재기동 시점에서는 남아있는 스냅샷 키를 먼저 처리한 뒤 새 RENAME을 수행해야 순서가 보존됩니다."

`outbox 패턴의 Redis판` + `멱등성 키`로 프레이밍하면 답변이 훨씬 단단해진다.

## 3. 서버 재기동과 고아 스냅샷

### Q. 단일 인스턴스인데 배포 시 두 인스턴스가 잠깐 겹치지 않나?

사이드 프로젝트라 자원이 부족해 두 인스턴스를 동시에 못 띄운다. 서버를 끄고 새 서버를 띄우는 방식. 그 사이에는 프론트 캐싱이나 기본값으로 대체.

### Q. 서버를 끄고 새 서버를 띄우는 사이 Redis 데이터는?

Redis는 별도 프로세스라 그대로 살아있다.

### Q. Schedule 실행 중에 서버가 꺼지면?

`graceful shutdown`이 있는 걸로 안다. 다만 이 프로젝트에는 적용 안 돼 있다. 놓친 부분.

### Q. DB 저장 도중 서버가 꺼지면 상태는?

3단계 중 어느 지점이냐에 따라:
- 1단계 RENAME 전 종료 → 원본 key에 데이터 그대로
- 2단계 DB 저장 중 종료 → 스냅샷만 남고 DB 미반영
- 3단계 스냅샷 삭제 직전 종료 → DB에 저장됐지만 스냅샷 잔존 (배치 ID로 skip)

### Q. 새 서버가 뜰 때 남아있는 스냅샷 키를 어떻게 감지/처리?

매 스케줄 시작 시 **패턴 기반으로 모든 스냅샷 키 조회** → DB에 해당 배치 ID 저장 로그가 있으면 스킵+삭제, 없으면 저장+삭제.

### 보완 포인트

답변은 흐름이 정확하다. 한 줄 보강:

> "새 서버가 뜰 때 `snapshot:*` 패턴으로 잔존 스냅샷을 먼저 처리하는 startup hook을 두면, 서버 재기동이 바로 복구 이벤트가 됩니다. Spring Boot의 `ApplicationReadyEvent`에 연결하면 scheduler가 돌기 전에 먼저 한 번 드레인할 수 있습니다."

`ApplicationReadyEvent` 같은 스프링 구체 훅 이름 하나만 박아두면 "설계 수준"에서 "구현 수준"으로 올라간다.

## 4. SCAN vs KEYS — 회복한 구간

### Q. 패턴 조회 명령어?

첫 답변: "Redis 명령어는 잘 몰라서 답변이 힘들다."

→ 면접관이 질문 방향을 바꾸자 차차 답이 나옴:

### Q. Hash 자료구조에서 field를 원자적으로 증가시키는 명령어?

처음 답변: `increment`

### Q. Hash field 증가 명령어가 INCR과 같은가?

Hash는 앞에 `H` 접두사가 붙는 걸로 안다 → **HINCRBY**.

### Q. 키가 많을 때 패턴 조회의 문제는?

Redis에는 인덱스가 없어 풀 스캔이 일어나 성능이 저하될 수 있다. (`{}`로 감싸면 특정 슬롯에 고정된다는 답변을 했다가 즉시 정정 — hash tag는 클러스터 노드 슬롯 고정 용도이지 인덱싱이 아니다.)

해결:
- **SCAN**으로 커서 기반 순회
- 또는 키 목록을 별도 Set으로 관리해 직접 조회

### Q. KEYS 대신 SCAN을 써야 하는 이유?

- **KEYS**: 한 번에 모든 키를 훑어 Redis의 싱글 스레드를 점유 → 다른 명령이 블록되어 장애 유발 가능
- **SCAN**: 커서 기반으로 나눠서 반환 → 다른 작업 블로킹 최소화

### 보완 포인트

처음엔 "모른다"로 완전히 무너졌다가 질문 흐름을 따라가며 SCAN/KEYS/HINCRBY까지 회복한 구간이 이번 세션의 가장 중요한 전환점이다. 보완 라인:

> "운영 환경에서 패턴 조회는 무조건 SCAN, 그리고 SCAN에도 `COUNT` 힌트를 적절히 줘서 round-trip을 조절합니다. 잔존 스냅샷이 매우 많아질 가능성이 있으면 Redis Set에 스냅샷 키 자체를 별도로 관리하는 방식이 SCAN보다 더 확실한 O(1) 조회가 됩니다."

`COUNT 힌트`, `Set으로 키 관리` 두 가지를 박아두면 SCAN 답변이 한 단계 올라간다.

### 다음 면접 전 Redis 필수 명령어 표

| 용도 | 명령어 |
|------|--------|
| String 증가 | `INCR`, `INCRBY` |
| Hash field 증가 | `HINCRBY` |
| 원자적 rename | `RENAME` (dest 없으면), `RENAMENX` (dest 있으면 실패) |
| 패턴 조회 (운영 금지) | `KEYS` |
| 패턴 조회 (운영 OK) | `SCAN`, `HSCAN`, `SSCAN` |
| 분산 락 | `SET ... NX EX`, Redlock |
| TTL | `EXPIRE`, `PEXPIRE` |

이 표를 외우지 않으면 면접에서 "명령어 모른다"가 나오고, 그 한 줄이 세션 전체 신뢰를 깎는다.

## 5. Summary 테이블 스키마와 인덱스

### Q. Summary 테이블 스키마?

조회수, 좋아요, 팔로우, 싫어요의 총량 + 업데이트 날짜.

### Q. "이번 주 조회수 추이" 같은 기능이 가능한가?

지금 구조로는 불가능. 주간 인기 정치인 페이지는 별도로 Redis에 일간 조회수를 모아두고 TTL 2주로 합산해 보여주고 있다. 과거 추이는 제대로 못 본다.

### Q. 날짜별 추이를 지원하려면?

총량 테이블 + **일별 테이블 + 월별 테이블**로 분리.

### Q. 일별 테이블 DDL 수준으로?

```
id PK
politician_id
view_count, like_count, follow_count, dislike_count
date
```

인덱스: PK 외에 날짜로 잡는다 — **정정** 필요. 조회 조건이 `politician_id + date`이므로 **복합 인덱스 (politician_id, date)**.

### Q. 복합 인덱스 순서의 근거?

예시:
- `(date, politician_id)` 순서: 날짜 범위 안의 모든 정치인 데이터를 먼저 추린 뒤 정치인 필터
- `(politician_id, date)` 순서: 정치인 단일 매칭으로 이미 좁아진 범위 안에서 날짜 범위 스캔

정치인 카디널리티가 작고, 날짜는 누적되어 시간이 지날수록 날짜 쪽 카디널리티가 더 커진다. 따라서 `(politician_id, date)`가 유리.

### 보완 포인트

이 답변은 이전 세션의 "데이터 조회 성능 최적화" 인덱스 답변과 동일한 근거를 사용하고 있다. 다음 면접 전 **복합 인덱스 순서의 답변 라인**을 고정하자:

> "복합 인덱스 순서는 leftmost prefix 규칙과 카디널리티 기반으로 정합니다. (1) 조건절에서 단일 값으로 매칭되는 컬럼(= politician_id)을 앞에 두어 후보를 빠르게 좁히고, (2) 범위 스캔이 필요한 컬럼(= date)을 뒤에 두어 B-Tree의 연속 영역을 그대로 읽습니다. '시간이 지나면 날짜 쪽이 더 커진다'는 부연은 맞지만 핵심 근거는 '단일 매칭 vs 범위 스캔'입니다."

`leftmost prefix + 단일 매칭 vs 범위 스캔`이 한 문장으로 고정되면 모든 인덱스 질문이 같은 답으로 수렴한다.

## 6. Spring Schedule + 멀티 인스턴스

### Q. 인스턴스가 여러 대면 어떤 문제?

각 인스턴스가 독립적으로 1분 타이머를 돌려 **중복 실행**이 발생. `Quartz`, `ShedLock` 같은 분산 락으로 해결.

### Q. 현재는 왜 적용 안 했나?

사이드 프로젝트라 인스턴스가 2개 올라갈 일이 없어서.

### 보완 포인트

"2개 올라갈 일이 없어서"는 배포 시점에 실제로 순간적으로 겹치는 케이스가 있으므로 정확하지 않다. 답변 라인:

> "현재는 단일 인스턴스 + 중단 배포라 분산 락이 필수는 아닙니다. 다만 배포 순간 두 프로세스가 겹치거나, 단일 인스턴스에서 스케줄 실행이 다음 주기까지 오버런되는 경우 중복 실행 위험이 있어, ShedLock을 적용하면 Redis나 DB 하나로 가볍게 막을 수 있습니다."

ShedLock + 단일 인스턴스도 선언해두는 게 면접에서 안전하다.

## 전체 평가

### 강점

- **Redis 장애 대응을 용도별로 분리** — 캐시 / 분산 락 / Stream으로 나눠서 답변
- **비즈니스 영향 기반 판단** — "조회수는 핵심 비즈니스가 아니다"라는 명확한 기준
- **원자성 흐름 분석** — RENAME의 원자성과 전체 흐름의 비원자성을 정확히 분리해서 설명
- **배치 ID 즉석 설계** — 면접관 질문을 받고 그 자리에서 멱등성 키 설계를 꺼냄
- **인덱스 순서 재정립** — 첫 답변(날짜만)을 면접관 지적으로 즉시 정정하고 근거까지 설명
- **SCAN 회복** — "Redis 명령어 모른다"에서 대화 흐름을 따라 SCAN/KEYS/HINCRBY까지 회복

### 약점 패턴

- **"Redis 명령어 잘 모른다"** — 핵심 인프라의 기본 명령어를 모른다는 답변은 면접에서 치명적. 대화로 회복은 됐지만 첫 답변의 타격이 크다
- **`{}` 해시 태그 오해** — 클러스터 노드 슬롯 고정 용도를 "인덱싱/순서"로 잘못 알고 있다가 즉시 정정
- **graceful shutdown 미적용** — 이력서의 "자동 복구" 문구와 실제 구현 사이의 갭. 면접에서 이런 갭이 가장 쉽게 드러난다
- **ShedLock 미적용** — "2개 띄울 일 없어서"로 일축하는 답변은 배포 순간의 겹침을 간과한 것
- **"ai 도움을 받아 해결하겠다"** — 면접관 앞에서 해결 도구로 "AI에 물어본다"는 답을 하는 건 위험. "공식 문서 + 실제 운영 사례 벤치마크"로 바꿔야 함
