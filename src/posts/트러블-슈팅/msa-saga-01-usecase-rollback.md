MSA 환경에서 서비스 간 통신이 늘어나면서, 하나의 요청이 여러 서비스에 걸쳐 상태를 변경해야 하는 상황이 빈번해졌다. 문제는 중간 단계에서 실패가 발생했을 때 이전 단계를 어떻게 되돌릴 것인가였다.

## 유스케이스 내부에 롤백 로직을 직접 작성하던 시기

처음에는 각 유스케이스 안에서 롤백 로직을 직접 정의했다. 예를 들어 "주문 생성" 유스케이스라면, GPU 할당 요청 후 실패 시 앞서 생성한 주문을 취소하는 코드를 유스케이스 내부에 함께 작성하는 방식이었다.

그런데 서비스 간 호출이 3단계, 4단계로 깊어지면서 유스케이스 클래스 하나에 정상 흐름과 각 단계별 롤백 분기가 뒤섞여 클래스의 복잡도가 과도하게 커졌다. 더 큰 문제는 동일한 실행 순서를 가져야 하는 유스케이스임에도 구현하는 사람마다 호출 순서나 롤백 범위가 달라져, 동일한 비즈니스 흐름에 대해 일관성이 깨지는 상황이 반복된 것이었다.

## Template 클래스 도입으로 순서 고정

이를 해결하기 위해 Template 클래스를 만들어 실행 순서를 내부에서 고정적으로 가져가도록 했다. 각 단계는 Outport의 결과만을 받아 다음 로직을 처리하고, 실패 시 역순으로 보상 트랜잭션을 실행하는 구조였다.

순서 불일치 문제는 해소되었지만, 롤백 도중 또다시 실패가 발생하면 어디까지 롤백이 완료되었고 어디서 멈췄는지 트래킹하기 어려웠다. 동기 호출 체인 안에서 보상 트랜잭션까지 동기로 처리하다 보니, 롤백 실패가 곧 전체 흐름의 상태 불명으로 이어졌다. 이 시점에서 동기 호출 기반의 롤백 방식에는 구조적 한계가 있다고 판단하고, 이벤트 기반 비동기 통신으로 전환하기로 결정했다.
