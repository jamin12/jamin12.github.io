## Redis Pub/Sub 시도와 한계

Kafka나 RabbitMQ가 이벤트 기반 아키텍처에 적합했지만, 당시 인프라 리소스 제한으로 별도의 메시지 브로커를 운영하기 어려운 상황이었다. 이미 캐시 용도로 사용 중이던 Redis를 활용하기로 하고, 먼저 Pub/Sub으로 이벤트를 처리했다.

Redis Pub/Sub은 구독자가 없거나 메시지를 수신하지 못하는 순간 메시지가 유실되는 구조였다. 서비스가 재시작되거나 네트워크 순단이 발생하면 그 사이의 이벤트는 복구할 방법이 없었다. 또한 Pub/Sub은 브로드캐스트 방식이라, 서비스가 스케일 아웃되면 동일한 이벤트를 모든 인스턴스가 수신하여 중복 처리가 발생하는 문제도 있었다.

## Redis Stream으로 전환

이 두 가지 문제를 해결하기 위해 Redis Stream으로 전환했다. Stream은 메시지가 로그처럼 영속적으로 저장되어 구독자가 없어도 유실되지 않고, Consumer Group을 통해 여러 인스턴스 중 하나만 메시지를 처리하도록 할 수 있어 스케일 아웃 환경에서의 중복 처리 문제를 해결할 수 있었다. 처리 완료 후 ACK를 보내는 구조라 실패한 메시지의 재처리도 가능했다.

## Outbox 패턴 도입

Redis Stream을 쓰더라도 근본적인 문제가 남아 있었다. 비즈니스 로직에서 DB 트랜잭션을 커밋한 뒤 Redis에 이벤트를 발행하는 사이에 애플리케이션이 죽으면, DB에는 상태가 변경되었지만 이벤트는 발행되지 않는 불일치가 발생할 수 있었다.

이를 방지하기 위해 Outbox 패턴을 도입했다. 이벤트를 Redis에 직접 발행하는 대신, 비즈니스 트랜잭션과 같은 DB 트랜잭션 안에서 Outbox 테이블에 이벤트를 함께 저장한다. 이후 별도의 스케줄러가 Outbox 테이블을 폴링하여 미발행 이벤트를 Redis Stream에 Push하는 구조다. DB 트랜잭션이 롤백되면 Outbox 레코드도 함께 롤백되므로, 비즈니스 상태와 이벤트 발행의 원자성이 보장된다.

## Outbox 테이블을 Status와 Event로 분리한 이유

Outbox 테이블 설계 과정에서 가장 많이 고민한 부분은, 하나의 테이블에 모든 정보를 담을 것인가 아니면 역할에 따라 분리할 것인가였다.

처음에는 단일 테이블로 시작했지만, 곧 두 가지 성격이 완전히 다른 데이터가 섞여 있다는 것을 알게 되었다. 하나는 "이 비즈니스 흐름이 지금 어떤 단계에 있는가"이고, 다른 하나는 "이 메시지가 Redis에 발행되었는가"였다. 전자는 비즈니스 맥락이고 후자는 인프라 맥락이다. 이 둘의 생명주기와 관리 방식이 근본적으로 달랐기 때문에 테이블을 분리하기로 결정했다.

**Outbox Status 테이블**은 비즈니스 흐름의 상태를 추적한다. 예를 들어 "주문 생성 → GPU 할당 → 과금 시작"이라는 흐름에서, 현재 어떤 단계까지 완료되었고 어떤 단계에서 실패했는지를 기록한다. 이 데이터는 영구적으로 보존되어야 했다. 장애가 발생했을 때 "이 주문은 GPU 할당까지 완료되었고 과금에서 실패했다"는 사실을 언제든 조회할 수 있어야 하기 때문이다. Status 테이블에는 흐름 식별자, 현재 단계, 각 단계의 성공/실패 여부, 실패 사유 등 비즈니스 판단에 필요한 데이터가 저장된다.

**Outbox Event 테이블**은 메시징 인프라의 발행 상태를 관리한다. 각 레코드는 "이 이벤트를 Redis Stream에 보내야 한다"는 발행 작업 단위이며, PENDING(미발행), SENT(발행 완료), FAILED(발행 실패) 같은 메시징 상태를 가진다. 이 데이터는 발행이 완료되면 더 이상 필요하지 않다. 스케줄러가 PENDING 상태의 레코드를 조회해 Redis에 Push하고 SENT로 변경한 뒤, 일정 기간이 지나면 정리할 수 있는 성격의 데이터다. Event 테이블에는 대상 Stream 이름, 페이로드, 발행 상태, 재시도 횟수 등 메시징 처리에 필요한 데이터가 저장된다.

만약 이 둘을 하나의 테이블에 두었다면, 발행 완료된 이벤트를 정리할 때 비즈니스 흐름 이력까지 함께 삭제되거나, 반대로 이력 보존을 위해 발행 완료된 레코드를 계속 쌓아두면서 스케줄러의 폴링 성능이 저하되는 문제가 생겼을 것이다. 분리함으로써 Event 테이블은 가볍게 유지하면서 빠른 폴링이 가능하고, Status 테이블은 비즈니스 이력으로서 온전히 보존되는 구조를 만들 수 있었다.
