## 중앙 오케스트레이션의 복잡성 폭발

Outbox 패턴으로 이벤트 발행의 안정성을 확보한 뒤, 본격적으로 Saga 패턴을 구현했다. 처음 선택한 방식은 중앙 오케스트레이션이었다. 중앙 레지스트리에 14개 비즈니스 흐름을 정의해두고, 라우터가 이벤트 타입을 보고 해당 처리기로 분배하고, 처리기가 비즈니스 로직을 수행한 뒤 다음 이벤트를 발행하는 구조였다.

동작은 했지만, 규모가 커지면서 문제가 드러났다. 처리기가 71개, 라우터가 14개, 구독자가 17개로 늘어났고, 보상 전용 조정기와 완료 처리기 같은 전용 인프라까지 더해지면서 Saga 전용 파일만 70개가 넘었다.

더 근본적인 문제는 이 구조가 기존에 쓰고 있던 헥사고날 아키텍처에서 완전히 벗어나 있다는 것이었다. Saga 처리기는 기존 유스케이스와 거의 동일한 일을 하면서도 인터페이스 없이 Saga 전용 인프라를 직접 의존하는 독자적 구조였다. 비즈니스 로직 2~5줄을 위해 Saga 인프라 보일러플레이트가 15~20줄 붙어 있었고, 이 구조를 이해하려면 기존 REST API 패턴과는 전혀 다른 개념을 새로 익혀야 했다. 혼자 만들고 혼자 유지보수하는 것이라면 모를까, 다른 팀원들도 함께 써야 하는 코드라는 점에서 이 복잡성은 치명적이었다.

## 기존 아키텍처로 전환

REST Controller가 HTTP 요청을 받아 유스케이스를 호출하듯, 구독자가 이벤트를 받아 유스케이스를 호출하는 구조로 되어 있었다. 외부 입력을 받는 어댑터만 다를 뿐, 그 이후의 흐름은 동일했다.

이 패턴을 Saga에 적용하면, 전용 처리기라는 독자적 구조를 만들 필요 없이 기존 유스케이스 패턴 그대로 비동기 이벤트 처리가 가능하다는 것을 깨달았다. 핵심 원칙은 단순했다.

- **동기 API**: HTTP 요청 → 유스케이스 → 외부 연동
- **비동기 Saga**: 이벤트 수신 → 유스케이스 → 외부 연동

중앙에서 흐름을 제어하는 레지스트리와 라우터를 제거하고, 각 구독자가 이벤트 타입에 따라 해당 유스케이스를 호출한 뒤, 성공/실패에 따라 다음 이벤트를 Outbox에 저장하는 방식이다.

## 흐름 제어의 책임을 구독자로 옮긴 이유

리팩터링에서 가장 중요한 설계 결정은 "다음 스텝을 누가 알고 있을 것인가"였다. 기존에는 처리기가 비즈니스 로직을 수행한 뒤 직접 다음 이벤트를 발행했다. 비즈니스 로직을 담당하는 코드가 "다음 스텝이 뭔지"까지 알아야 했고, 이것이 비즈니스 로직과 흐름 제어 로직이 뒤섞이는 원인이었다.

전환한 구조에서는 유스케이스가 순수 비즈니스 로직만 담당하고, 구독자(외부 입력 어댑터)가 흐름을 소유한다. REST Controller가 유스케이스 호출 결과를 보고 "어떤 HTTP 응답을 내릴지" 결정하는 것과 동일하게, 구독자가 유스케이스 실행 결과를 보고 "어떤 다음 이벤트를 발행할지" 결정하는 것이다. 이렇게 하면 유스케이스에 이벤트 발행 의존성이 주입될 필요가 없어지고, 기존 REST API용 테스트 패턴을 그대로 사용할 수 있게 된다.

## 정방향과 보상을 구분하지 않는 단일 구조

이벤트 타입을 `{STEP}_SUCCESS`와 `{STEP}_FAILED`로 설계하면서, 정방향과 보상을 별도로 구분할 필요가 없어졌다. 구독자 입장에서는 어떤 이벤트가 들어오든 동일한 방식으로 처리한다. 성공 이벤트가 들어오면 다음 유스케이스를 실행하고, 실패 이벤트가 들어오면 보상 유스케이스를 실행한다. 둘 다 "이벤트 타입을 받고, 매핑된 유스케이스를 실행하고, 결과 이벤트를 발행한다"라는 하나의 규칙으로 동작한다.

각 스텝은 자기가 체인의 어디에 있는지 알지 못한다. 이전 스텝이 뭔지, 다음 스텝이 뭔지 모른다. 자기가 받은 이벤트에 매핑된 유스케이스를 실행하고, 그 결과로 성공 또는 실패 이벤트를 발행할 뿐이다. 보상 전용 조정기나 완료 처리기 같은 별도 인프라가 필요하지 않다.

다만 보상 유스케이스는 멱등하게 구현했다. 보상 대상이 존재하지 않으면 에러를 던지지 않고 아무것도 하지 않는 no-op으로 처리한다. 실제로 수행되지 않은 스텝의 보상이 호출되거나 네트워크 재시도로 중복 호출되더라도 안전하게 처리된다.

## 결과

이 리팩터링으로 Saga 전용 인프라 약 70개 파일이 제거되었다. 중앙 레지스트리, 라우터 14개, 보상 조정기와 완료 처리기가 사라지고, 전용 처리기 71개가 기존 패턴과 동일한 유스케이스로 전환되었다. 구독자는 17개에서 3개로 줄었다.

가장 중요한 성과는 신규 팀원이 분산 트랜잭션 코드를 이해하기 위해 새로운 개념을 배울 필요가 없어진 것이다. 기존에는 레지스트리, 라우터, 전용 처리기, 보상 조정기 등 Saga 전용 개념 세트를 별도로 익혀야 했지만, 전환 후에는 이벤트 구독자, 유스케이스, 외부 연동 등 이미 REST API 개발에서 매일 쓰고 있는 것들만 알면 된다. 동기 API 패턴을 이해하는 사람이라면 분산 트랜잭션 코드도 바로 읽을 수 있는 구조가 된 것이다.

## 이후

이 리팩터링 직후의 결론은 단순화였지만, 이후 운영하면서 단순 2테이블 / 단일 이벤트 규칙만으로는 부족한 영역이 드러났다. 보상 유스케이스의 멱등(no-op)만으로는 일반 Step의 중복 처리 방지가 모자랐고, Status 한 컬럼으로 에러를 추적하다가 보상 실패가 원본 실패를 덮어쓰는 사례가 발생했으며, ACK만 신뢰하기에는 응답 유실·서버 다운에 대한 안전망이 부족했다. 이 한계를 반영해 테이블·재시도·라우팅 모델을 다시 정교화했다.
