서버가 실시간으로 데이터를 밀어내는 방식은 몇 가지가 있다. HTTP 롱 폴링, Server-Sent Events (SSE), WebSocket. 각각 특성이 달라서 우리 요구에 맞는 걸 골라야 했다.

## 우리가 필요로 하는 것

- 서버가 능동적으로 클라이언트에 데이터 push
- **한 연결에서 여러 스트림을 동시에** 다룰 수 있을 것 (한 화면에서 Pod 로그 + 이벤트 같이)
- 구독 시작/해제 제어를 클라이언트에서 가능
- 긴 수명 (로그 시청 시간이 분~시간 단위)
- 확장 여지: 나중에 **양방향 스트림** (웹 터미널 pod exec) 같은 걸 붙일 수 있어야 함

## 선택지 비교

**HTTP 롱 폴링**은 클라이언트가 요청을 보내고 서버가 값이 생길 때까지 응답을 지연시키는 방식이다. 구현은 간단하지만 본질은 반복 요청이라 우리가 폴링에서 벗어나려던 의도와 어긋난다. 길어진 요청을 서버가 물고 있어야 하고, 프록시나 로드밸런서의 타임아웃에 걸릴 위험도 있다.

**Server-Sent Events**는 HTTP 위에서 서버가 일방적으로 이벤트 스트림을 보내는 표준이다. 브라우저에서 `EventSource` API 로 다룬다. 서버→클라 단방향만 지원하고, 구독 시작은 URL, 해제는 연결을 닫는 것으로만 가능하다. 다중 구독을 하려면 **구독마다 연결을 새로** 열어야 한다.

**WebSocket**은 HTTP 핸드셰이크로 시작해서 양방향 프레임 통신으로 업그레이드되는 프로토콜이다. 한 연결 위에서 양쪽이 자유롭게 메시지를 주고받는다. 메시지 포맷도 우리가 정의한다. 연결 하나로 구독 여러 개를 다중화하기에 가장 자연스럽다.

| 항목 | 롱 폴링 | SSE | WebSocket |
|------|---------|-----|-----------|
| 전송 방향 | 양방향(반복) | 서버→클라 | 양방향 |
| 다중 스트림 | ✗ | ✗ (연결 N개) | ✓ |
| 양방향 확장 (웹 터미널) | 번거로움 | 불가 | 쉬움 |
| 구독 해제 제어 | 재요청 필요 | 연결 close 외엔 어려움 | 메시지로 명시 |

"한 연결 + 다중 구독 + 양방향 확장 여지" 세 가지를 동시에 만족시키는 건 WebSocket 뿐이었다.

## WebSocket 위에서 또 한 번: Raw vs STOMP

WebSocket 을 고르면 또 하나의 갈림길이 있다. 프로토콜을 **맨바닥부터 JSON 으로 설계**할지, 아니면 Spring 이 기본 지원하는 **STOMP** 를 쓸지다.

STOMP 는 pub/sub 모델을 가진 프레임 지향 프로토콜이다. 클라이언트가 `/topic/...` 을 SUBSCRIBE 하면 서버가 해당 토픽에 메시지를 발행할 때 **구독자 모두에게** 뿌려진다. 채팅, 알림처럼 한 메시지를 여러 구독자에게 broadcast 하는 시나리오에 잘 맞는다.

우리 요구는 정반대다. "로그인한 사용자가 요청한 Pod 로그를 **그 사용자 한 명한테만** 전달" 하는 1:1 스트림이다. 구독자 간 공유나 broadcast 가 전혀 필요 없다. 이 경우 STOMP 의 pub/sub 레이어는 쓰이지 않는 오버헤드다. 토픽 이름 규약, 메시지 변환기, 메시지 브로커 구성 같은 개념들이 의미 없이 복잡도를 더한다.

Raw WebSocket + JSON 프로토콜 하나만 직접 정의하면 충분하다. 필요한 의미만 담는다.

```json
// 구독 시작
{ "action": "subscribe", "subscriptionId": "...", "type": "pod-log", "params": { ... } }

// 구독 해제
{ "action": "unsubscribe", "subscriptionId": "..." }

// 서버가 보내는 데이터
{ "subscriptionId": "...", "type": "data", "payload": "..." }
```

Spring 이 기본 지원해주는 STOMP 의 편의를 포기하는 대신, 매 필드가 왜 필요한지 설명할 수 있는 형태가 된다.

## 단일 엔드포인트 + type 라우팅

처음엔 도메인별로 엔드포인트를 나누는 방식도 생각했다. `/ws/pod-log`, `/ws/pipelinerun-log` 처럼. 이 방식은 직관적이지만 다섯 개 화면 × 앞으로 추가될 기능을 곱하면 엔드포인트 수가 계속 늘어난다. 새 스트림 종류를 붙일 때마다 엔드포인트와 핸들러를 새로 등록해야 한다.

단일 엔드포인트로 두고 subscribe 메시지의 `type` 필드로 라우팅하는 방식을 택했다. `/ws/stream` 하나만 있고, 클라이언트가 `type: "pod-log"` 로 subscribe 하면 서버가 내부에서 적절한 스트림 구현체를 찾아 연결한다. 새 기능을 붙일 때는 **type 이름을 하나 정하고 구현체를 등록** 하는 것으로 끝난다.

## 결정 요약

- 프로토콜: **Raw WebSocket + 커스텀 JSON**
- 엔드포인트: 단일 `/ws/stream`
- 다중화: subscribe 메시지의 `subscriptionId` 로 스트림 구분, `type` 으로 도메인 라우팅
