자바·코틀린과 Go가 크게 다른 부분이 `*`와 `&`다. 자바는 객체를 무조건 참조로 다루고 그 "참조"를 JVM이 숨겨주지만, Go는 값으로 복사할지 참조로 가리킬지를 직접 고른다. 그 선택 도구가 `*`와 `&`다.

## `&`와 `*`는 정반대 방향이다

- **`&`** = 값의 주소를 얻는다 (값 → 포인터)
- **`*`** = 상황에 따라 두 가지 의미를 갖는다

```go
user := User{Name: "Koa"}   // 값
p := &user                  // & : user의 주소 → p는 포인터
v := *p                     // * : p가 가리키는 값을 꺼냄 → 다시 User
```

`&`로 포인터를 만들고 `*`로 다시 값을 꺼낸다. `&`는 의미가 하나뿐이라 쉽다. 문제는 `*`다.

## `*`의 두 가지 의미

`*`는 위치에 따라 의미가 갈린다.

```go
var p *User      // ① 타입 앞 → "User 포인터 타입"
v := *p          // ② 값 앞   → 역참조 (알맹이 꺼내기)
```

| `*`의 위치 | 의미 | 예시 |
|---|---|---|
| 타입 앞 | "~를 가리키는 포인터 타입" | `*User`, `func(u *Service)` |
| 값 앞 | 역참조 (알맹이 꺼내기) | `*p`, `*p = User{...}` |

실무에서 보는 `*`는 거의 다 **타입 자리**다. Go가 `p.Name`처럼 점 접근할 때 자동으로 역참조해 주기 때문에 `(*p).Name`을 쓸 일이 거의 없다. 그래서 코드에서 마주치는 패턴은 둘뿐이다. 타입 자리의 `*`(포인터 타입)와, 객체를 만들며 주소를 얻는 `&`다.

```go
func (u *BannerUsecase) Create() {}      // 리시버 타입 (포인터)
func NewBanner() *BannerUsecase {        // 반환 타입 (포인터)
    return &BannerUsecase{...}           // & : 객체 생성 + 포인터 반환
}
```

`&Struct{...}`는 자바의 `new Class(...)`라고 생각하면 거의 맞다.

## 값으로 넘기면 복사된다, 하지만 깊은 복사는 아니다

`*` 없이 값으로 넘기면 struct가 통째로 복사된다. 그래서 복사본을 수정해도 원본은 그대로다.

```go
func modify(u User) { u.Age = 99 }   // 복사본 수정 → 원본 영향 없음
```

문제는 이게 **얕은 복사(shallow copy)**라는 점이다. struct의 맨 위 필드만 복사되고, 그 안에 슬라이스·맵·포인터 같은 참조성 타입이 있으면 알맹이는 원본과 공유된다.

```go
type User struct {
    Name string      // 값 타입 → 독립 복사
    Tags []string    // 슬라이스 → 알맹이 공유
}

func modify(u User) {
    u.Name = "changed"     // ① 복사본만 바뀜
    u.Tags[0] = "hacked"   // ② 원본도 바뀜!
}

user := User{Name: "Koa", Tags: []string{"a", "b"}}
modify(user)
// user.Name → "Koa"    (① 안 바뀜)
// user.Tags[0] → "hacked"  (② 바뀜)
```

이유는 슬라이스의 정체에 있다. 슬라이스는 순수한 값이 아니라 "실제 배열을 가리키는 포인터 + 길이 + 용량"으로 된 작은 구조다. struct를 복사하면 이 포인터까지 복사되고, 포인터가 가리키는 실제 배열은 원본과 같은 것을 공유한다. `&`를 직접 쓰지 않았어도 슬라이스 안에 이미 포인터가 들어 있는 셈이다.

| 필드 타입 | 값 복사 시 |
|---|---|
| `int`, `string`, `bool`, 중첩 struct, 배열 `[N]T` | 독립 복사 |
| `slice`, `map`, `*T`, `chan` | 포인터만 복사 → 알맹이 공유 |

진짜 깊은 복사가 필요하면 직접 해야 한다. Go에는 자동 깊은 복사가 없다.

```go
newTags := make([]string, len(u.Tags))
copy(newTags, u.Tags)
```

## 포인터는 힙을 직접 다루는 게 아니다

C 경험이 있으면 "포인터 = 힙을 직접 할당하고 해제한다"로 오해하기 쉽다. Go는 자바처럼 GC가 있는 언어다. `malloc`/`free`가 없고, 메모리 해제는 전부 GC가 한다.

게다가 `&User{}`가 무조건 힙으로 가는 것도 아니다. Go 컴파일러는 **이스케이프 분석(escape analysis)**으로 "이 포인터가 함수 밖으로 새어나가는가"를 보고 스택에 둘지 힙에 둘지 자동으로 정한다.

```go
func sum() int {
    u := User{Age: 10}
    p := &u            // & 썼지만 함수 안에서만 살다 죽음 → 스택
    return p.Age
}

func NewUser() *User {
    u := User{Age: 10}
    return &u          // 함수 밖으로 escape → 힙
}
```

둘 다 `&`를 썼지만 하나는 스택, 하나는 힙이다. 개발자가 정하는 게 아니라 컴파일러가 정한다. 그러니 `*`는 메모리 위치를 고르는 도구가 아니라 **값 의미론과 참조 의미론을 고르는 도구**로 이해하는 게 맞다.

## 언제 포인터로 넘기나

실무 기준은 단순하다.

- **객체(서비스·핸들러·큰 struct)는 거의 항상 포인터.** 하나의 인스턴스를 공유해야 하고, 복사 비용을 피해야 하며, `sync.Mutex` 같은 걸 품으면 복사 시 깨지기 때문이다.
- **작고 읽기 전용인 DTO나 기본 값은 값으로.** 복사 비용이 적고 원본이 안 바뀌어 오히려 안전하다.

한 타입의 메서드 리시버는 포인터면 전부 포인터, 값이면 전부 값으로 통일하는 게 관례다. 그래서 서비스류는 `func (u *Service)`로 통일된다.

생성자는 거의 항상 `&`로 포인터를 반환한다.

```go
func NewBannerV2(cfg IConfig, repo IRepository) *BannerV2Usecase {
    return &BannerV2Usecase{config: cfg, repository: repo}
}
```

`*Type`을 "자바의 객체 참조", `&obj`를 "자바의 new"라고 대응시키면 대부분 맞는다.
