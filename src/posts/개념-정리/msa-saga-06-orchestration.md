## 1. 오케스트레이션 vs 코레오그래피

### 비교

| | 오케스트레이션 | 코레오그래피 |
|--|--------------|------------|
| 다음 Step 결정 | 중앙 오케스트레이터 | 각 서비스/Consumer가 직접 |
| 흐름 파악 | 한 곳에서 전체 보임 | 여러 서비스 따라가야 함 |
| 결합도 | 오케스트레이터가 모든 서비스 알아야 함 | 서비스 간 독립적 |
| 디버깅 | 쉬움 (중앙 상태) | 어려움 (이벤트 추적 필요) |
| 보상 | 중앙에서 역순 실행 | 각 서비스가 알아서 |

코레오그래피가 무조건 좋은 건 아니다. 실무에서는 오케스트레이션을 더 많이 쓴다. 코레오그래피는 단순할 때 좋지만 복잡해지면 이벤트 흐름 추적이 지옥이 된다.

### 하이브리드 형태

실무에서는 순수한 한쪽만 쓰기보다 두 방식을 섞는 경우가 많다. 예를 들어 상태 관리는 중앙화(Saga State로 추적)하면서, 흐름 제어는 분산(각 Consumer가 다음 Step을 직접 지정)하는 형태다.

이런 하이브리드가 실용적인 이유는 중앙 상태 추적의 디버깅 이점은 가져가면서, 각 서비스가 자기 비즈니스 흐름을 직접 제어할 수 있기 때문이다.

### 순수 오케스트레이션의 형태

순수 오케스트레이션에서는 참여자가 결과만 응답하고, 오케스트레이터가 흐름 정의에서 다음 Step을 자동 결정한다.

```kotlin
// 참여자: 결과만 응답
"결제 성공했어"

// 오케스트레이터: 흐름 정의에서 다음 Step 자동 결정
class OrderSagaOrchestrator {
    val flow = listOf(
        Step("PAYMENT", paymentStream),
        Step("INVENTORY", inventoryStream),
    )
    fun onStepCompleted(sagaId, stepName, result) {
        val nextStep = flow.nextAfter(stepName)
        publish(nextStep, ...)
    }
}
```

## 2. 오케스트레이션에서의 메시지 흐름

### 참여자는 결과만 응답한다

오케스트레이션에서 참여자는 전체 흐름을 모른다. 결과만 보내고, 다음 Step 판단은 오케스트레이터의 몫이다.

``` text
X 참여자 B: "결제 성공했으니 재고 차감 이벤트 발행할게"
   → B가 전체 흐름을 알아야 함 → 이건 코레오그래피

O 참여자 B: "결제 성공했어" (응답만)
   → 오케스트레이터 A: "그럼 다음은 재고 차감이네" → Outbox INSERT
```

다른 서비스로 넘어가면 다음 Step 이벤트가 아니라 **결과 응답**을 Outbox에 넣는 것이다.

### reply_to 패턴

오케스트레이터가 메시지를 보낼 때 응답 채널 정보를 포함해야 참여자가 결과를 돌려보낼 수 있다.

```json
{
  "saga_id": "uuid-1234",
  "saga_type": "ORDER_CREATE",
  "reply_to": "saga:order:reply",
  "step": "PAYMENT_REQUEST",
  "payload": { ... }
}
```

## 3. Saga vs Long Running Process

### Saga가 해결하려는 원래 문제

순수 Saga는 **분산 DB 간 트랜잭션 일관성**을 위한 패턴이다.

- **순수 Saga**: 서비스 A DB + 서비스 B DB + 서비스 C DB → 각 서비스가 이벤트를 수신하고 자기 DB를 변경

하지만 내가 진행 중인 프로젝트는 DB 1개에 외부 API(Keycloak, K8s 등)를 호출하는 구조였다. 분산 DB 간 일관성이 아니라, 실패 시 보상 추적과 비동기 처리가 핵심 가치였다.

### 이벤트 기반 vs API 호출

순수 Saga가 성립하려면 상대도 이벤트를 수신하고 처리하는 서비스여야 한다. 외부 API(Keycloak, K8s)는 이벤트를 받지 않으니 직접 API를 호출해야 한다.

| | 이벤트 (비동기) | API 호출 (동기) |
|--|--------------|---------------|
| 상대 다운 시 | 브로커에 쌓여있다가 살아나면 처리 | 즉시 실패 |
| 결합도 | 낮음 | 높음 |

**핵심 질문은 "상대 서비스가 죽어있어도 괜찮아야 하나?"**다. YES라면 이벤트 기반, NO라면 API 호출로 충분하다.

### DB 1개 + 외부 API 구조에서 Saga의 가치

| 가치 | 해당 여부 |
|------|----------|
| 분산 DB 일관성 | X (DB 1개) |
| 실패 시 보상 추적 | O (핵심) |
| 비동기 처리 (긴 작업 분리) | O |
| Step별 상태 관리 | O |

이런 구조는 정확히는 **Long Running Process** 또는 **Process Manager** 패턴에 해당한다. 이름만 다르지 구현은 거의 같다.
