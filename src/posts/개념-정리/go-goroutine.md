Go의 동시성은 `go` 키워드 하나로 시작한다. 자바에서 스레드를 다뤄봤다면 개념은 익숙하지만, 고루틴이 스레드와 어떻게 다르고 왜 수십만 개를 만들어도 괜찮은지는 따로 이해해야 한다. 이 글은 고루틴의 정체와 Go 런타임의 스케줄링 모델을 정리한다.

## go 키워드

`go f()`는 그 함수를 고루틴으로, 즉 백그라운드에서 동시에 실행하라는 뜻이다.

```go
r.Run()      // 끝날 때까지 여기서 멈춤 (블로킹)
go r.Run()   // 백그라운드로 던지고 즉시 다음 줄로 (논블로킹)
```

자바의 `new Thread(() -> ...).start()`에 해당하지만 키워드 하나로 끝난다. 다만 두 가지를 주의해야 한다. 첫째, `main`이 끝나면 고루틴도 같이 죽는다. 그래서 서버처럼 계속 살아야 하는 경우 종료 신호 대기 같은 장치로 `main`을 붙잡아 둔다. 둘째, `go`로 던진 함수의 반환값이나 에러는 그 자리에서 못 받는다. 받으려면 채널(channel)로 결과를 보내야 한다.

```go
ch := make(chan int)
go func() { ch <- compute() }()   // 결과를 채널로 보냄
result := <-ch                     // 채널에서 받음
```

서버 코드에서는 보통 여러 서버를 동시에 띄울 때 쓴다. HTTP 서버, gRPC 서버, 메시지 컨슈머를 각각 `go`로 올리고 `main`은 종료 신호를 기다린다.

## 고루틴은 실제 실행 단위다

고루틴은 추상적 개념이 아니라 런타임이 메모리에 실제로 만드는 경량 실행 단위다. `go f()`를 실행하면 런타임이 고루틴 하나를 생성하고, 각 고루틴은 자기만의 스택(처음 약 2KB, 필요 시 자동으로 커짐)과 상태(실행 중·대기·잠듦)를 가진다.

| 단위 | 관리 주체 | 크기 |
|---|---|---|
| 프로세스 | OS | 무거움 |
| 스레드 | OS | 무거움 (MB) |
| 고루틴 | Go 런타임 | 가벼움 (~2KB) |

스레드가 실제 단위인 것처럼 고루틴도 실제 단위다. 단지 OS가 아니라 Go 런타임이 관리하고 훨씬 가볍다. `runtime.NumGoroutine()`으로 살아 있는 개수를 셀 수 있고, 패닉 스택 트레이스에도 `goroutine 18 [chan receive]`처럼 번호와 상태가 찍힌다.

## M:N 스케줄링 — 고루틴 多, 스레드 少

핵심은 수많은 고루틴을 소수의 OS 스레드 위에 얹어 돌린다는 점이다. 고루틴이 100만 개여도 OS 스레드는 보통 CPU 코어 수 수준이다.

Go 런타임은 이걸 G-M-P 모델로 관리한다.

- **G (goroutine)**: 작업 단위. 멈췄다 재개되는 대상.
- **M (machine)**: 실제 OS 스레드. 진짜 일꾼.
- **P (processor)**: 스케줄러. 실행 가능한 고루틴 큐를 관리하며 개수는 `GOMAXPROCS`(보통 코어 수).

P가 "실행 가능한 고루틴 목록"을 들고, M(스레드)이 P에 붙어 고루틴을 하나씩 집어 실행한다. 어떤 고루틴이 멈추면 M은 그 고루틴을 내려놓고 다음 고루틴을 집는다. 즉 스레드가 고루틴을 빌리는 게 아니라, **스레드가 실행 가능한 고루틴을 계속 집어 가며 일한다.**

요리사(스레드)가 주문(고루틴)을 처리하다가 재료를 기다려야 하면 그 주문을 잠깐 제쳐두고 다른 주문을 처리하는 것과 같다.

## 멈췄다 재개되는 원리

고루틴이 멈출 때(예: I/O 대기) 자기 상태(스택, 실행 위치)를 통째로 저장하고 스레드에서 내려온다. 그 스레드는 다른 고루틴을 실행하고, 나중에 그 고루틴이 재개될 때는 아무 스레드 위에서나 저장해 둔 상태를 복원해서 멈췄던 곳부터 이어간다.

그래서 스레드는 고루틴이 실제로 CPU를 쓸 때만 점유되고, 멈춘 고루틴은 스레드 없이 메모리에 상태로만 누워 있는다. 이것이 소수의 스레드가 수많은 고루틴을 번갈아 실행할 수 있는 이유다.

## 스레드는 미리 잔뜩 만들지 않는다

오해하기 쉬운 부분인데, 고루틴이 많다고 스레드를 미리 많이 만들어 두지 않는다. 기본은 코어 수 수준이고, 스레드가 늘어나는 건 **블로킹 syscall이 생길 때뿐**이다.

고루틴이 파일 읽기 같은 블로킹 시스템 콜을 호출하면 그 스레드가 묶이므로, 런타임은 CPU를 놀리지 않기 위해 다른 스레드를 추가로 만들어 나머지 고루틴을 계속 돌린다. 작업이 끝나면 그 스레드는 회수·재사용된다. 미리가 아니라 막힐 때 그때그때 만든다.

## 네트워크 대기는 스레드를 쓰지 않는다

Go의 강력한 특징이 여기 있다. 외부 API 호출처럼 네트워크 응답을 기다릴 때는 스레드를 점유하지 않는다. **netpoller**라는 장치 덕분이다.

```
고루틴이 외부 API 호출 → 응답 대기
  → 고루틴을 재워두고(park) 스레드는 놓아줌 (다른 고루틴 실행에 씀)
  → OS의 epoll/kqueue가 "응답 도착" 알리면
  → 고루틴을 깨워 이어서 실행
```

그래서 외부 API 호출 1만 개가 동시에 대기 중이어도 스레드는 여전히 몇 개뿐이다. 기다리는 고루틴은 스레드 없이 잠들어 있을 뿐이다. Go가 적은 자원으로 높은 동시성을 버티는 비결이다.
