Kubernetes 기반 플랫폼을 운영하면서 특정 Pod 의 컨테이너 로그를 실시간으로 보여줘야 하는 화면이 다섯 개였다.

- Workload > Pod 상세 > 컨테이너 로그
- CI/CD > PipelineRun 상세 > TaskRun step 로그
- CI/CD > 파이프라인 실행 > 실행이력 > 로그
- CI/CD > 파이프라인 실행 목록 > 마지막 실행 로그
- GitOps > 배포 애플리케이션 상세 > Pod 실행 로그

지금까지는 REST 로 단발 조회를 하는 구조였다. `PodK8sRepository.getPodContainerLog(...)` 가 Fabric8 의 `logs()` 를 호출해서 그 시점까지의 로그를 스냅샷으로 가져온다. 사용자가 화면을 띄워두면 FE 가 일정 간격으로 같은 엔드포인트를 반복 호출해서 마치 실시간처럼 보이게 만들었다.

## 폴링 구조의 한계

단발 조회로 실시간을 흉내 내려면 FE 가 결국 **폴링**을 돌릴 수밖에 없다. 이 구조는 두 지점에서 부담이 된다.

첫 번째는 **반복 요청의 비용**이다. 한 번의 로그 조회가 가볍지 않다. BE 가 K8s API 서버에 HTTPS 요청을 보내고, API 서버가 kubelet 으로부터 로그를 모아 응답하는 과정이다. Pod 한 대의 로그를 5초마다 요청하는 사용자가 열 명이면 K8s API 서버에 초당 2건, 분당 120건의 로그 요청이 꾸준히 쌓인다. 대시보드 특성상 한 사용자가 오래 띄워두는 경우가 많아 실질적인 부담은 더 크다.

두 번째는 **끊기는 실시간성**이다. 폴링은 본질적으로 스냅샷의 반복이라, 로그가 쏟아지는 순간과 화면에 보이는 순간 사이에 폴링 간격만큼의 지연이 생긴다. 배포 직후의 기동 로그나 에러 폭발 순간처럼 **바로 봐야 하는 시점**일수록 이 지연이 눈에 띈다. 간격을 좁히면 첫 번째 비용이 다시 커진다.

세 번째는 **중복 라인의 처리**다. 폴링은 매 회마다 "처음부터 tailLines 만큼" 을 다시 받는다. FE 가 직전에 그린 라인과 방금 받은 라인을 비교해 새로 추가된 부분만 붙여야 하는데, 컨테이너가 재시작되거나 로그가 로테이션되면 이 비교가 깨진다. "끊김 없이 이어지는 로그" 를 FE 쪽에서 재구성해야 하는 부담이 누적된다.

## 서버가 밀어주는 방향으로

FE 가 계속 가져가는 구조에서 서버가 능동적으로 밀어주는 구조로 바꾸면 세 문제가 동시에 풀린다. 연결을 한 번 수립하면 로그가 생성되는 순간마다 그 라인만 전달하면 되므로, K8s API 호출 횟수는 **구독 건수만큼**으로 떨어지고, 지연은 네트워크 왕복 시간까지 줄어든다. 새로 붙는 라인만 내려보내니 FE 도 들어오는 값을 그대로 append 하면 된다.

서버가 밀어주는 방식은 몇 가지 선택지가 있다. HTTP 롱 폴링, Server-Sent Events, WebSocket. 각각 특성이 다르고 우리 요구사항에 따라 맞는 게 갈린다. 비교와 선택 이유는 다음 편에서 정리한다.

## 이번 범위

작업의 범위는 **실시간 로그 스트리밍 인프라를 새로 세우고, pod-log 한 가지 도메인을 먼저 붙이는 것**까지다. 나머지 네 화면은 PipelineRun step 로그, 파이프라인 실행 로그 등 도메인이 다르지만, 인프라 위에 **구현체 하나 추가** 로 동작해야 한다는 전제를 깔고 설계한다.

권한 검증 세분화, 여러 Pod 로그를 한 화면에서 동시에 다루는 UI 최적화, 재연결 시 직전 구독을 복원하는 로직 같은 주제는 여기서는 다루지 않는다. 핵심은 "실시간 스트림" 자체를 가능하게 만드는 것이고, 그 위에 얹을 기능은 후속으로 빼둔다.

다음 편에서 WebSocket 을 고른 이유, 그리고 왜 STOMP 를 쓰지 않고 Raw WebSocket + JSON 프로토콜로 갔는지 이어간다.
