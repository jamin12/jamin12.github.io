서킷브레이커를 개념으로만 알면 "좋은 거니까 다 붙이면 되겠네?"라고 생각하기 쉬웠다. 실제로는 아니었다.

## 판단 기준

서킷브레이커를 붙일지 말지는 보통 세 가지를 본다.

- 외부 의존성인가
- 반복 실패 가능성이 있는가
- 계속 붙잡고 있으면 내 쪽도 위험해지는가

이 세 가지가 맞으면 서킷브레이커 후보가 된다.

## 가장 대표적인 적용 대상: 원격 호출

외부 HTTP API, 다른 마이크로서비스 호출, gRPC 호출, 결제/문자/메일/지도/인증 같은 외부 SaaS 연동이 제일 전형적이다. 내 프로세스 밖에 있고, 네트워크를 타고, 상대 상태를 내가 통제할 수 없기 때문이다. 원래부터 느려질 수 있고, 실패할 수 있고, 일시 장애든 지속 장애든 생길 수 있다.

예를 들어 주문 처리 중에 외부 결제 승인 API를 호출하는 상황이라고 해보자. 결제사가 장애가 나면 승인 요청이 오래 걸리고, timeout이 쌓이고, 주문 처리 스레드가 묶이고, 전체 주문 응답이 느려진다. 서킷브레이커가 있으면 최근 결제 승인 실패가 계속 누적될 때 당분간 결제사 호출을 차단해서 내 주문 서비스까지 같이 무너지는 걸 막을 수 있다.

## DB에는 붙이나?

DB도 외부 시스템이긴 하다. 그런데 무조건 서킷브레이커부터 떠올리진 않았다. DB는 보통 내 서비스의 핵심 의존성이라서, 차단해봤자 서비스가 아예 의미를 잃는 경우가 많기 때문이다. 주문 서비스가 주문 DB를 못 읽는데 서킷브레이커로 DB 호출을 막아버리면, 결국 서비스 자체가 거의 못 돌아간다.

읽기 캐시가 충분하고, 일부 비핵심 조회가 DB 실패 시 캐시나 기본값으로 대체 가능하다면 부분적으로 고려할 수는 있었다. 하지만 일반적으로 DB 앞에서는 connection pool 보호, timeout, 쿼리 최적화, read replica나 캐시 전략 같은 것들을 먼저 봤다.

## 같은 프로세스 내부 메서드 호출에는?

보통 안 붙인다. 서비스 클래스 내부 메서드 호출, util 함수 호출, 같은 JVM 안의 계산 로직 같은 것들은 네트워크 원격 호출이 아니고, 호출 비용도 작고, 상태 전이도 빠르다. 여기서 실패는 대개 외부 장애가 아니라 버그, 입력값 문제, 로직 문제다. 서킷브레이커는 "상대가 불안정해서 잠깐 차단하자"는 철학인데, 내부 메서드 버그는 차단한다고 해결되지 않는다.

금액 계산, DTO 변환, 문자열 파싱, 정렬/집계 같은 순수 계산 로직도 마찬가지다. 여기서 문제가 나면 코드 수정, 예외 처리, 입력 검증이 필요한 것이지 "당분간 계산 안 함"은 이상한 대응이었다.

## 캐시 조회

상황을 봐야 했다. Redis가 외부 인프라라면 네트워크 장애나 지연이 생길 수 있고, 계속 Redis timeout이 쌓이면 애플리케이션도 느려질 수 있다. 비핵심 캐시 조회에는 서킷브레이커를 고려할 수 있었다. 추천 캐시 조회 실패 시 추천 없음 처리, 최근 본 상품 캐시 실패 시 빈 목록 처리 같은 식이다. 차단 후 대체가 가능한 경우였다.

반면 Redis가 사실상 핵심 시스템이고 없으면 전체 동작이 안 되는 구조라면 서킷브레이커 하나로 해결되지 않았다.

## 메시지 브로커, 큐, 이벤트 발행

비동기 알림 발행, 로그성 이벤트 발행, 분석 이벤트 전송처럼 실패해도 핵심 비즈니스는 계속 가야 하는 영역이면 서킷브레이커가 잘 맞았다. 잠깐 차단해도 핵심 트랜잭션 전체를 깨지 않고 버틸 수 있기 때문이다.

반면 정말 핵심적인 메시지 송신이면 단순 차단보다 outbox 같은 신뢰성 설계가 더 중요했다. 메시징 쪽은 서킷브레이커가 보조 수단일 수는 있지만 전달 보장 문제를 대신 해결해주진 않는다.

## 비핵심 부가 기능에 특히 잘 맞는다

서킷브레이커가 가장 잘 맞는 곳은 실패해도 핵심 플로우는 유지되어야 하는 부가 기능이었다. 추천, 리뷰 집계, 통계 수집, 사용자 행동 분석, 알림, 서드파티 부가 정보 조회 같은 영역이다. 외부 서비스가 잠깐 죽어도 핵심 주문, 로그인, 조회 같은 본 기능은 살려두고 싶었다. 서킷브레이커로 차단하고 fallback으로 빈값이나 캐시, 기본값을 반환하는 조합이 잘 먹혔다.

## 핵심 기능에는?

붙일 수는 있었다. 다만 더 신중해야 했다. 로그인에 필요한 외부 인증 서버, 주문에 필수인 결제 승인, 핵심 데이터 조회용 서비스라면 차단은 가능하지만 그 결과가 매우 크다. 이런 곳에 서킷브레이커를 붙인다는 건 "일정 조건이면 이 핵심 기능을 잠시 실패시키겠다"는 뜻이기도 했다.

그래서 fallback이 가능한지, partial degradation이 가능한지, 정말 차단하는 게 전체 장애보다 나은지, 재시도나 timeout이나 bulkhead와 조합은 어떻게 할 건지를 같이 생각해야 했다. 핵심 기능일수록 설정이 더 보수적이어야 했다.

## 붙일지 고민될 때의 질문

어떤 호출 앞에 서킷브레이커를 붙일지 고민될 때 이 질문들을 해봤다.

1. 이 호출은 원격 의존성인가? 네트워크, 외부 인프라, 외부 시스템인가?
2. 여기서 지연이나 실패가 반복될 수 있는가?
3. 계속 시도하면 내 자원이 고갈될 수 있는가? 스레드, 커넥션, 응답시간, 큐 적체에 영향이 큰가?
4. 잠깐 차단했을 때 감당 가능한가? fallback, 캐시, 빈값, 부분 실패 처리라도 가능한가?

많이 맞을수록 서킷브레이커 후보였다. 반대로 안 맞는 쪽의 공통점은 외부 의존성이 아니거나, 장애가 아니라 코드 버그 성격이거나, 차단해도 보호 효과가 없거나, 차단 시 서비스 의미가 거의 사라지거나, fallback이 전혀 말이 안 되는 경우였다.

## 결국 봐야 할 것

서킷브레이커는 실패를 숨기는 도구가 아니라 외부 장애가 내 시스템 전체로 번지는 걸 막는 도구였다. "실패가 나올 수 있냐"만 볼 게 아니라, 그 실패를 계속 끌고 가면 내 쪽이 얼마나 위험해지는가를 같이 봐야 했다.
