MSA 환경에서 서비스 간 통신이 늘어나면서, 하나의 요청이 여러 서비스에 걸쳐 상태를 변경해야 하는 상황이 빈번해졌다. 문제는 중간 단계에서 실패가 발생했을 때 이전 단계를 어떻게 되돌릴 것인가였다.

## 유스케이스 내부에 롤백 로직을 직접 작성하던 시기

처음에는 각 유스케이스 안에서 롤백 로직을 직접 정의했다. 예를 들어 "주문 생성" 유스케이스라면, GPU 할당 요청 후 실패 시 앞서 생성한 주문을 취소하는 코드를 유스케이스 내부에 함께 작성하는 방식이었다.

그런데 서비스 간 호출이 3단계, 4단계로 깊어지면서 유스케이스 클래스 하나에 정상 흐름과 각 단계별 롤백 분기가 뒤섞여 클래스의 복잡도가 과도하게 커졌다. 더 큰 문제는 동일한 실행 순서를 가져야 하는 유스케이스임에도 구현하는 사람마다 호출 순서나 롤백 범위가 달라져, 동일한 비즈니스 흐름에 대해 일관성이 깨지는 상황이 반복된 것이었다.

## Template 클래스 도입으로 순서 고정

이를 해결하기 위해 Template 클래스를 만들어 실행 순서를 내부에서 고정적으로 가져가도록 했다. 각 단계는 Outport의 결과만을 받아 다음 로직을 처리하고, 실패 시 역순으로 보상 트랜잭션을 실행하는 구조였다.

순서 불일치 문제는 해소되었지만, 롤백 도중 또다시 실패가 발생하면 어디까지 롤백이 완료되었고 어디서 멈췄는지 트래킹하기 어려웠다. 동기 호출 체인 안에서 보상 트랜잭션까지 동기로 처리하다 보니, 롤백 실패가 곧 전체 흐름의 상태 불명으로 이어졌다. 이 시점에서 동기 호출 기반의 롤백 방식에는 구조적 한계가 있다고 판단하고, 이벤트 기반 비동기 통신으로 전환하기로 결정했다.

## Redis Pub/Sub 시도와 한계

Kafka나 RabbitMQ가 이벤트 기반 아키텍처에 적합했지만, 당시 인프라 리소스 제한으로 별도의 메시지 브로커를 운영하기 어려운 상황이었다. 이미 캐시 용도로 사용 중이던 Redis를 활용하기로 하고, 먼저 Pub/Sub으로 이벤트를 처리했다.

Redis Pub/Sub은 구독자가 없거나 메시지를 수신하지 못하는 순간 메시지가 유실되는 구조였다. 서비스가 재시작되거나 네트워크 순단이 발생하면 그 사이의 이벤트는 복구할 방법이 없었다. 또한 Pub/Sub은 브로드캐스트 방식이라, 서비스가 스케일 아웃되면 동일한 이벤트를 모든 인스턴스가 수신하여 중복 처리가 발생하는 문제도 있었다.

## Redis Stream으로 전환

이 두 가지 문제를 해결하기 위해 Redis Stream으로 전환했다. Stream은 메시지가 로그처럼 영속적으로 저장되어 구독자가 없어도 유실되지 않고, Consumer Group을 통해 여러 인스턴스 중 하나만 메시지를 처리하도록 할 수 있어 스케일 아웃 환경에서의 중복 처리 문제를 해결할 수 있었다. 처리 완료 후 ACK를 보내는 구조라 실패한 메시지의 재처리도 가능했다.

## Outbox 패턴 도입

Redis Stream을 쓰더라도 근본적인 문제가 남아 있었다. 비즈니스 로직에서 DB 트랜잭션을 커밋한 뒤 Redis에 이벤트를 발행하는 사이에 애플리케이션이 죽으면, DB에는 상태가 변경되었지만 이벤트는 발행되지 않는 불일치가 발생할 수 있었다.

이를 방지하기 위해 Outbox 패턴을 도입했다. 이벤트를 Redis에 직접 발행하는 대신, 비즈니스 트랜잭션과 같은 DB 트랜잭션 안에서 Outbox 테이블에 이벤트를 함께 저장한다. 이후 별도의 스케줄러가 Outbox 테이블을 폴링하여 미발행 이벤트를 Redis Stream에 Push하는 구조다. DB 트랜잭션이 롤백되면 Outbox 레코드도 함께 롤백되므로, 비즈니스 상태와 이벤트 발행의 원자성이 보장된다.

## Outbox 테이블을 Status와 Event로 분리한 이유

Outbox 테이블 설계 과정에서 가장 많이 고민한 부분은, 하나의 테이블에 모든 정보를 담을 것인가 아니면 역할에 따라 분리할 것인가였다.

처음에는 단일 테이블로 시작했지만, 곧 두 가지 성격이 완전히 다른 데이터가 섞여 있다는 것을 알게 되었다. 하나는 "이 비즈니스 흐름이 지금 어떤 단계에 있는가"이고, 다른 하나는 "이 메시지가 Redis에 발행되었는가"였다. 전자는 비즈니스 맥락이고 후자는 인프라 맥락이다. 이 둘의 생명주기와 관리 방식이 근본적으로 달랐기 때문에 테이블을 분리하기로 결정했다.

**Outbox Status 테이블**은 비즈니스 흐름의 상태를 추적한다. 예를 들어 "주문 생성 → GPU 할당 → 과금 시작"이라는 흐름에서, 현재 어떤 단계까지 완료되었고 어떤 단계에서 실패했는지를 기록한다. 이 데이터는 영구적으로 보존되어야 했다. 장애가 발생했을 때 "이 주문은 GPU 할당까지 완료되었고 과금에서 실패했다"는 사실을 언제든 조회할 수 있어야 하기 때문이다. Status 테이블에는 흐름 식별자, 현재 단계, 각 단계의 성공/실패 여부, 실패 사유 등 비즈니스 판단에 필요한 데이터가 저장된다.

**Outbox Event 테이블**은 메시징 인프라의 발행 상태를 관리한다. 각 레코드는 "이 이벤트를 Redis Stream에 보내야 한다"는 발행 작업 단위이며, PENDING(미발행), SENT(발행 완료), FAILED(발행 실패) 같은 메시징 상태를 가진다. 이 데이터는 발행이 완료되면 더 이상 필요하지 않다. 스케줄러가 PENDING 상태의 레코드를 조회해 Redis에 Push하고 SENT로 변경한 뒤, 일정 기간이 지나면 정리할 수 있는 성격의 데이터다. Event 테이블에는 대상 Stream 이름, 페이로드, 발행 상태, 재시도 횟수 등 메시징 처리에 필요한 데이터가 저장된다.

만약 이 둘을 하나의 테이블에 두었다면, 발행 완료된 이벤트를 정리할 때 비즈니스 흐름 이력까지 함께 삭제되거나, 반대로 이력 보존을 위해 발행 완료된 레코드를 계속 쌓아두면서 스케줄러의 폴링 성능이 저하되는 문제가 생겼을 것이다. 분리함으로써 Event 테이블은 가볍게 유지하면서 빠른 폴링이 가능하고, Status 테이블은 비즈니스 이력으로서 온전히 보존되는 구조를 만들 수 있었다.

## 중앙 오케스트레이션의 복잡성 폭발

Outbox 패턴으로 이벤트 발행의 안정성을 확보한 뒤, 본격적으로 Saga 패턴을 구현했다. 처음 선택한 방식은 중앙 오케스트레이션이었다. 중앙 레지스트리에 14개 비즈니스 흐름을 정의해두고, 라우터가 이벤트 타입을 보고 해당 처리기로 분배하고, 처리기가 비즈니스 로직을 수행한 뒤 다음 이벤트를 발행하는 구조였다.

동작은 했지만, 규모가 커지면서 문제가 드러났다. 처리기가 71개, 라우터가 14개, 구독자가 17개로 늘어났고, 보상 전용 조정기와 완료 처리기 같은 전용 인프라까지 더해지면서 Saga 전용 파일만 70개가 넘었다.

더 근본적인 문제는 이 구조가 기존에 쓰고 있던 헥사고날 아키텍처에서 완전히 벗어나 있다는 것이었다. Saga 처리기는 기존 유스케이스와 거의 동일한 일을 하면서도 인터페이스 없이 Saga 전용 인프라를 직접 의존하는 독자적 구조였다. 비즈니스 로직 2~5줄을 위해 Saga 인프라 보일러플레이트가 15~20줄 붙어 있었고, 이 구조를 이해하려면 기존 REST API 패턴과는 전혀 다른 개념을 새로 익혀야 했다. 혼자 만들고 혼자 유지보수하는 것이라면 모를까, 다른 팀원들도 함께 써야 하는 코드라는 점에서 이 복잡성은 치명적이었다.

## 기존 아키텍처로 전환

REST Controller가 HTTP 요청을 받아 유스케이스를 호출하듯, 구독자가 이벤트를 받아 유스케이스를 호출하는 구조로 되어 있었다. 외부 입력을 받는 어댑터만 다를 뿐, 그 이후의 흐름은 동일했다.

이 패턴을 Saga에 적용하면, 전용 처리기라는 독자적 구조를 만들 필요 없이 기존 유스케이스 패턴 그대로 비동기 이벤트 처리가 가능하다는 것을 깨달았다. 핵심 원칙은 단순했다.

- **동기 API**: HTTP 요청 → 유스케이스 → 외부 연동
- **비동기 Saga**: 이벤트 수신 → 유스케이스 → 외부 연동

중앙에서 흐름을 제어하는 레지스트리와 라우터를 제거하고, 각 구독자가 이벤트 타입에 따라 해당 유스케이스를 호출한 뒤, 성공/실패에 따라 다음 이벤트를 Outbox에 저장하는 방식이다.

## 흐름 제어의 책임을 구독자로 옮긴 이유

리팩터링에서 가장 중요한 설계 결정은 "다음 스텝을 누가 알고 있을 것인가"였다. 기존에는 처리기가 비즈니스 로직을 수행한 뒤 직접 다음 이벤트를 발행했다. 비즈니스 로직을 담당하는 코드가 "다음 스텝이 뭔지"까지 알아야 했고, 이것이 비즈니스 로직과 흐름 제어 로직이 뒤섞이는 원인이었다.

전환한 구조에서는 유스케이스가 순수 비즈니스 로직만 담당하고, 구독자(외부 입력 어댑터)가 흐름을 소유한다. REST Controller가 유스케이스 호출 결과를 보고 "어떤 HTTP 응답을 내릴지" 결정하는 것과 동일하게, 구독자가 유스케이스 실행 결과를 보고 "어떤 다음 이벤트를 발행할지" 결정하는 것이다. 이렇게 하면 유스케이스에 이벤트 발행 의존성이 주입될 필요가 없어지고, 기존 REST API용 테스트 패턴을 그대로 사용할 수 있게 된다.

## 정방향과 보상을 구분하지 않는 단일 구조

이벤트 타입을 `{STEP}_SUCCESS`와 `{STEP}_FAILED`로 설계하면서, 정방향과 보상을 별도로 구분할 필요가 없어졌다. 구독자 입장에서는 어떤 이벤트가 들어오든 동일한 방식으로 처리한다. 성공 이벤트가 들어오면 다음 유스케이스를 실행하고, 실패 이벤트가 들어오면 보상 유스케이스를 실행한다. 둘 다 "이벤트 타입을 받고, 매핑된 유스케이스를 실행하고, 결과 이벤트를 발행한다"라는 하나의 규칙으로 동작한다.

각 스텝은 자기가 체인의 어디에 있는지 알지 못한다. 이전 스텝이 뭔지, 다음 스텝이 뭔지 모른다. 자기가 받은 이벤트에 매핑된 유스케이스를 실행하고, 그 결과로 성공 또는 실패 이벤트를 발행할 뿐이다. 보상 전용 조정기나 완료 처리기 같은 별도 인프라가 필요하지 않다.

다만 보상 유스케이스는 멱등하게 구현했다. 보상 대상이 존재하지 않으면 에러를 던지지 않고 아무것도 하지 않는 no-op으로 처리한다. 실제로 수행되지 않은 스텝의 보상이 호출되거나 네트워크 재시도로 중복 호출되더라도 안전하게 처리된다.

## 결과

이 리팩터링으로 Saga 전용 인프라 약 70개 파일이 제거되었다. 중앙 레지스트리, 라우터 14개, 보상 조정기와 완료 처리기가 사라지고, 전용 처리기 71개가 기존 패턴과 동일한 유스케이스로 전환되었다. 구독자는 17개에서 3개로 줄었다.

가장 중요한 성과는 신규 팀원이 분산 트랜잭션 코드를 이해하기 위해 새로운 개념을 배울 필요가 없어진 것이다. 기존에는 레지스트리, 라우터, 전용 처리기, 보상 조정기 등 Saga 전용 개념 세트를 별도로 익혀야 했지만, 전환 후에는 이벤트 구독자, 유스케이스, 외부 연동 등 이미 REST API 개발에서 매일 쓰고 있는 것들만 알면 된다. 동기 API 패턴을 이해하는 사람이라면 분산 트랜잭션 코드도 바로 읽을 수 있는 구조가 된 것이다.
